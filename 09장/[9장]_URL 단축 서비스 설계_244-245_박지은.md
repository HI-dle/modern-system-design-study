## 9.1.1 기능적 요구 사항

URL 단축 서비스를 설계하는 과정에서의 여러 가지 기능적 요구 사항

- 핵심
  - 긴 URL을 입력받으면 단축 URL을 반환해야 한다. 
  - 단축 URL을 입력받으면 원래의 긴 URL을 반환해야 한다. 
  - 단축 URL은 고유해야 하며 길이도 일정해야 한다.
- 추가 고려사항
  - 기능적 요구 사항
    - 입력받은 긴 URL의 유효성을 검사
    - 사용자가 맞춤형 URL을 생성할 수 있도록 지원
    - 단축 URL은 마지막 사용 시점을 기준으로 6개월 후 만료
    - 오류가 발생했을 때도 시스템이 정상 동작 처리
    -  URL 생성자는 긴 URL을 수정 가능 해야함.
    -  URL 매핑을 삭제할 수 있는 기능을 제공.
    - 단축 URL로 활용도 분석 및 모니터링이 가능해야 한다.
    - 사용자 계정을 관리할 수 있다.
## 9.1.2. 비기능적 요구 사항
대규모 시스템에서의 비기능적 요구사항 
- 가용성: 시스템이 항상 안정적 동작
- 확장성: 사용자 1억 명 이상을 지원, 요청이 증가하더라도 정상 대응
- 응답 속도: 읽기와 쓰기 요청을 지연 없이 처리
- 일관성: 
  - 동일한 단축 URL로 긴 URL에 접근하려는 두 사용자에게는 동일한 내용의 원본 URL 반환
  - 동일한 긴 URL에 대해 단축 URL을 생성하려는 두 사용자에게는 동일한 내용의 단축 URL을 반환.
  - 한 사용자가 특정 긴 URL에 대한 단축 URL을 생성한 후 다시 같은 긴 URL로 단축 URL을 생성하면 동일한 단축 URL 생성
- 내구성: 단축 URL이 생성된 후에는 데이터가 삭제되거나 손실 없음. 즉, 시스템 장애 및 예기치 못한 상황에서도 데이터를 안전하게 유지.
- 신뢰성: 시스템에 장애, 요청 폭주, 기타 문제가 생기더라도 올바르게 작동해야 하며, 요구 사항을 충족.
