## 방법 3: 카운터 방식
숫자를 하나씩 늘려서 고유한 값을 만드는 방식. 
### ex.
처음에 1부터 시작해서 다음 요청이 들어올 때마다 2, 3, 4처럼 숫자를 하나씩 증가.

이렇게 생성된 숫자를 기반으로 단축 URL을 만들어 내면 충돌이 발생할 가능성은 없다.

### 한 서버가 이 숫자(카운터)를 관리
- 요청이 들어올 때마다 새로운 숫자를 발급
  - ex. short.ly/1, short.ly/2, short.ly/3
- 문제점: 고유 ID를 생성하는 서버가 단일 장애 지점(Single Point Of Failure, SPOF)이 됨
  - 서버에 문제가 생기거나 다운되면 시스템 전체가 ID를 생성하지 못하게 되어 단축 URL 서비스가 중단될 수 있다.

### 여러  서버가 카운터 관리
서버를 여러 대 사용하면 단일 장애 지점 문제를 피할 수 있지만, 각 서버에서 생성하는 카운터 값을 고유하게 만드는 것은 쉽지 않다. 
- 호스트 ID(6비트), 시간 ID(32비트), 카운터 증가 값(4비트)을 조합 하는 방법으로 해결할 수도 있다.
- 문제
  - 호스트 ID에 6비트를 할당하면 호스트를 최대 64대만 사용 가능
  - 마지막 4비트는 증가하는 카운터를 대비한 용도로, 1밀리초에 요청이 16개 초과하면 충돌 발생
    - 호스트 수가 64대로 제한되므로 대규모 시스템에서는 확장성이 떨어짐. 
    - 1밀리초 동안 요청이 16개를 초과하면 충돌 발생 위험 ➡️ 높은 트래픽 상황에서 작동 어려움
    
## 방법 4: 분산 카운터 방식

시퀀스 번호 3.5조 개를 여러 범위로 나누는 방식. 

각 범위는 k라고 가정하며, k를 100만(1M)으로 설정할 경우 번호를 3.5조 개 처리하는 데 범위가 3.5백만 개 필요(각 범위가 숫자를 100만 개 포함)

-  범위 350만 개
  - ex. R1 [0-1M), R2 [1M-2M), R3 [9M-10M)
    - R1은 첫 번째 범위로 0부터 100만 미만까지 숫자
    - R2는 100만부터 200만 미만까지 숫자를 포함
- 이런 범위를 고가용성 서버(범위 할당 서버(range server)라고 하자)에 저장 가능. 
- 여러 카운터 서버가 범위 할당 서버와 통신하며 범위를 요청
- 범위 할당 서버는 요청을 받은 후 각 카운터 서버에 특정 범위를 할당하고, 다음과 같은 맵 구조로 저장
  - R1 [0-1M) → s1 
  - R2 [1M-2M) → s2 
  - R10 [9M-10M) → s10
- 카운터 서버는 자신에게 할당된 범위 내에서 숫자를 하나씩 증가시키며 고유한 숫자를 생성
  - ➡️ 여러 카운터 서버 간에도 중복 방지
- 만든 고유 숫자를 a-z, A-Z, 0-9 문자 집합을 사용하여 7자리 문자열로 변환 과정
  - 문자열 변환에 사용할 62진수 문자 집합은 다음과 같습니다. 
    - abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789 
  - ex. 고유 숫자: 9234529445
    - 10진수에서 62진수로 변환
      > 9234529445 62 = 148907051, 나머지 23 → x
      > 
      > 148907051 62 = 2401748, 나머지 39 → N
      > 
      > 2401748 62 = 38702, 나머지 4 → e
      > 
      > 38702 62 = 624, 나머지 14 → 0
      > 
      > 624 62 = 10, 나머지 4 → e
      > 
      > 10 : 62 = 0, 나머지 10 → k 
      - 카운터 숫자 9234529445를 Base 62로 변환하면 {10, 4, 14, 4, 39, 23}이 된다.
      - Base 62문자 세트에 매핑 ➡️ 'keoeNx' 
      - 최종 URL: urlshortener.com/keoeNx

- 문제 
  - 카운터 서버가 할당된 범위를 모두 사용하면?
    - 해당 카운터 서버는 범위 할당 서버에 새 범위를 요청한다. 
  - 카운터 서버가 다운되면? 
    - 할당된 범위는 그대로 버려져 해당 범위(100만 단위)는 손실. 
      - 전체 범위 350만 개 중 하나만 잃기에 크게 문제될 일은 아니다. 
      - 게다가 카운터 서버가 실패하는 경우는 드물기에 이 정도 손실은 감당할 수 있다. 
  - 범위를 모두 잃는 상황은?
    - 해결 방법은 카운터 서버가 주기적으로 범위 할당 서버에 상태를 기록. 
      -  ex. N번째 URL(N = 100 또는 1000)을 생성할 때마다 현재 카운터 값을 저장. 
        - ➡️ 서버가 중단되더라도 마지막으로 저장된 값에 N만큼 더한 위치에서 다시 시작할 수 있어 한 번의 장애로 최대 N개만 손실 
  - 범위 할당 서버가 단일 장애 지점 가능성
    - 범위 할당 서버는 읽기와 쓰기 요청이 많지 않아 고가용성, 복제 기능, 장애 허용성을 갖춘 주키퍼 같은 서비스를 사용하면 이 문제를 해결할 수 있다.
- 이런 문제점이 있을 수 있지만, 여전히 이전에 소개한 방법보다는 훨씬 견고해 보이며 이렇다 할 단점도 없다.
