# [9장]_URL 단축 서비스 설계

---

## 단축 URL 생성 방법

 - 단축 URL 서비스를 설계할 때 어떤 방법을 선택할 수 있는지, 어디서 병목이 발생하는지, 발생할 수 있는 문제와 대응 방법은 무엇인지, 전체 시스템 구조는 어떻게 되는지 살펴본다.
 - 가장 먼저 이 시스템에서 풀어야 할 핵심 문제부터 본다.

### 9.4.1 핵심 문제

 - URL 단축 서비스를 설계할 때 가장 중요한 문제는 "어떻게 하면 겹치지 않는 고유한 단축 URL을 효율적으로 생성할 것인가"이다.
 - 앞서 본 요구 사항과 규모를 생각하면 이 문제를 풀 방법은 여러 가지가 있다.
 - 각 방법의 장단점을 하나씩 분석해 보고 가장 적합한 해결책을 찾는다.

### 방법 1: 무작위로 URL 생성하기

 - 긴 URL을 입력받으면 무작위로 단축 URL을 생성한 뒤, DB에 해당 URL이 이미 있는지 확인한다.
 - 이미 있으면 새 URL을 다시 생성하고, 중복 검사를 통과할 때까지 이 과정을 반복한다.
 - 중복 검사를 통과한 단축 URL을 찾으면 키-값 쌍으로 DB에 저장한다.
   - {short_url -> long_url}: 단축 URL을 키로, 긴 URL을 값으로 묶어 저장한다.

 - 무작위 방식에는 몇 가지 문제가 있다.
   - 새로 만든 단축 URL이 DB에 이미 있는지 확인하려면 최소 한 번은 DB 조회가 필요하다. 경우에 따라 중복된 단축 URL이 생성되어 재생성 → 재조회 과정을 반복하게 된다. 이 과정이 반복되면 예측할 수 없는 지연이 생기고, 쓰기 요청의 지연 시간이 늘어난다.
   - 두 번째 문제는 동시성으로 인한 충돌이다. 요청 두 개가 동시에 같은 단축 URL을 생성하려는 상황을 가정한다. 요청 하나는 long_url1을, 다른 하나는 long_url2를 처리하는데, 우연히 두 요청이 같은 단축 URL을 만들어 낼 수 있다. 이 경우 long_url1의 쓰기가 성공한 뒤 long_url2가 그 데이터를 덮어써서 손상시킬 수 있다.
     - long_url1과 long_url2가 우연히 같은 short_url1로 만들어진다는 건, 요청 A({long_url1 → short_url1})와 요청 B({long_url2 → short_url1})가 같은 결과를 전달한다는 의미다.
     - 요청 A가 short_url1이 DB에 있는지 확인한다. 이 시점에 short_url1은 없지만, 그 사이에 요청 B도 short_url1이 비어 있다고 판단하고 {short_url1 → long_url2}를 저장한다.
 - 이후 요청 A가 다시 short_url1을 저장하면서 기존 항목을 덮어써 {short_url1 → long_url2}로 남게 된다. 이 때문에 long_url2는 손상되어 사라진다.

 - 이 문제를 어떻게 풀어야 하는가?
   - putIfAbsent 같은 기능을 쓰면 해결할 수 있다. 다만 모든 DB가 이 기능을 지원하는 건 아니다.
   - 다른 방법으로는 값을 DB에 저장한 뒤 getLongUrl(short_url)을 호출해, 해당 short_url에 대한 long_url이 요청한 값과 일치하는지 확인하는 방식이 있다. 이렇게 하면 데이터가 손상되지 않았는지 확인할 수 있다. 다만 성공할 때까지 검증이 여러 번 필요할 수 있고, 쓰기 요청 한 번마다 최소 한 번의 읽기 요청이 추가되어 비효율적이다.
 - 이 방식은 실제 운영 환경에서 쓸 만한 최적의 방법은 아니므로, 다른 대안을 더 살펴본다.

### 방법 2: MD5 해시

 - 긴 URL을 입력으로 받아 MD5 해시를 생성하는 방법이다.
 - MD5 해시는 128비트 길이를 가지며, 현재 예시 가정에서는 62진수 기반 7자리 단축 URL을 만들어야 한다.
   - 이를 위해서는 약 42비트가 필요하다(62^7 ≒ 2^42).
   - 따라서 128비트 길이의 MD5 해시에서 42비트를 가져와, 이를 7자리 62진수 문자로 변환하면 단축 URL을 생성할 수 있다.
 - 장점
   - 저장 공간을 효율적으로 쓸 수 있다. 동일한 긴 URL에 대한 요청이 다시 들어와도 같은 단축 URL을 생성하므로, 데이터를 중복 저장하지 않는다.
 - 문제점
   - 128비트 전체를 쓰지 않고 일부만 쓰기 때문에 해시 값의 충돌 확률이 크게 증가한다. URL을 MD5 해시로 변환했을 때 앞 42비트가 같을 가능성이 높기 때문이다.

> MD5 해시만 믿었다간 낭패 보는 상황
>
> MD5 해시는 128비트짜리 고유 값을 만들어 준다고 생각하지만, 여기서 문제는 "해시 일부만 가져오면?"이다. 128비트 중 앞 일부(예: 42비트)만 떼어 오면 충돌 확률이 확 높아진다. 즉, 서로 다른 URL인데도 똑같은 단축 URL이 생성되는 불상사가 생길 수 있다.

```javascript
const crypto = require('crypto');

function generateShortUrl(longUrl) {
  // MD5 해시를 생성하고 앞 일곱 글자만 사용
  const hash = crypto.createHash('md5').update(longUrl).digest('hex');
  return hash.substring(0, 7);  // 해시 앞 일곱 글자만 추출
}

const url1 = "https://example.com/user/profile?id=12345";
const url2 = "https://example.com/user/profile?id=67890";

const shortUrl1 = generateShortUrl(url1);
const shortUrl2 = generateShortUrl(url2);

console.log("URL 1 단축 결과:", shortUrl1);
console.log("URL 2 단축 결과:", shortUrl2);

// 충돌 체크
if (shortUrl1 === shortUrl2) {
  console.log("충돌 발생! 두 URL이 같은 단축 URL을 가져 버렸네요.");
} else {
  console.log("안전! 각 URL이 다른 단축 URL을 가집니다.");
}
```

> 코드를 돌려 보면 가끔 '우연히' 두 URL이 똑같은 단축 URL을 가지는 상황을 볼 수 있다. '우연히'라고 했지만 사실 이건 해시 일부만 사용해서 발생하는 필연적인 문제다. 마치 성씨만 보고 "이 사람 누구네 집안 사람이야!"라고 결론을 내는 격이다.
>
> 그러니 해시를 쓸 때는 전부를 신뢰하되, 일부만 쓰는 일은 조심해야 한다.

 - 42비트보다 더 많은 비트를 쓰면 충돌 확률은 낮아지지만, 그만큼 단축 URL 길이가 늘어난다. 이는 URL 단축 서비스의 핵심인 '짧은 URL'이라는 장점을 해친다.
 - 또 128비트를 전부 쓰지 않는 한 충돌은 완전히 피할 수 없다.
 - 결론적으로 이 방법도 완벽하지 않으므로, 다른 방법을 계속 살펴본다.
