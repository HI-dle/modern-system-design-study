# [9장]_URL 단축 서비스 설계

---


## 계산으로 문제 규모 파악

 - 단축 URL에 사용할 수 있는 문자 종류
   - a-z, A-Z, 0-9로 총 62개
   - 추가로 -, ., _, ~ 더하면 66개가 되지만 설명에서는 62개만 다룬다.
 - 서비스 규모
   - 전체 사용자 수: 약 10억 명
     - 이 중 10%인 1억 명이 단축  URL을 생성한다고 가정
     - 서비스하루 이용량: 1억건
     - 각 사용자가 하루에 URL을 한 개씩 만들어 총 1억 개를 만든다 가정하고 이 방식으로 URL을 10년간 저장한다고 하면 저장하는 총 URL 수는 1억 * 1 * 365 * 10 = 3650억 개가 된다.
 - 단축 URL 길이
   - 고유 URL을 3650억 개 만드려면 몇 글자를 사용해야 하는가?
   - 6글자를 사용했을때 URL을 62^6 = 500억 개 생성가능하지만 3650억에 부족하다.
   - 7글자를 사용했을때 URL을 62^7 = 약 3조 5000억 개 생성할 수 있다.
 - 필요한 저장 공간
   - URL을 총 3650억개 저장해야하니 시스템 내에 저장해야 하는 컬럼은
     - 단축 URL: 20 바이트
     - 원본 URL: 1000 바이트
     - 생성 시각: 10 바이트
     - 수정 시각: 10 바이트
     - 생성자: 20 바이트
   - 따라서 URL당 저장해야 하는 총 데이터 크기는 1060 바이트이고 이를 1500 바이트로 반올림 한다면 저장해야 하는 총량은 3650억 * 1500바이트로 500TB이다
   - 레플리카를 3개 사용할때 필요한 저장 용량은 약 1.5PB가 된다.
 - 읽기 및 쓰기 작업의 초당 요청 수(Request per Second, RPS)
   - 쓰기 RPS: 하루에 요청이 1억 개 발생하면 1억 / (86400초/일) = 1억 / 100,000(86400을 대략 10만으로 근사) = 1000RPS
   - 읽기 RPS: 일반적으로 읽기 요청은 쓰기 요청보다 100배 정도 많다고 가정하면 1000 RPS -100 = 100,000RPS
 - 이제까지의 계산으로 높은 읽기 쓰기 RPS를 처리하려면 여러대의 서버가 반드시 필요하고 URL의 규모를 감당하려면 단출 URL은 최소 7자리 문자를 사용해야 한다는 것도 확인했다.
