# 9.5 요구 사항 검토

이 장에서는 URL 단축 서비스를 만든다고 가정하고, 앞서 살펴본 설계가 **기능적 요구 사항**과 **비기능적 요구 사항**을 제대로 충족하는지 빠르게 점검한다.

### 기능적 요구 사항 검토

#### 1) 긴 URL을 입력받으면 단축 URL을 반환해야 한다

#### 2) 단축 URL을 입력받으면 원래의 긴 URL을 반환해야 한다

#### 3) 단축 URL은 고유해야 하며 길이도 일정해야 한다

- 설계된 방식에서는 단축 URL이 **고유(Unique)** 하며, 길이 또한 **일정한 7글자**로 유지된다.

### 비기능적 요구 사항 검토

#### 1) 가용성(Availability)

단축 URL 서비스는 항상 높은 가용성을 유지해야 한다.

- 시스템이 **수평 확장 구조**를 가짐
- **단일 장애 지점(SPOF, Single Point of Failure)** 이 없음
- 읽기/쓰기 서버를 분리하여 요청을 안정적으로 처리 가능

즉, 일부 서버에 장애가 발생하더라도 전체 서비스는 계속 동작할 수 있습니다.


#### 2) 확장성(Scalability)

시스템은 **1억 명 이상의 사용자**와 **요청 폭주 상황**도 감당할 수 있어야 한다.

이 설계에서는 다음 방식으로 확장성을 확보한다.

- **카운터 서버**와 **읽기 서버**를 로드 밸런서 뒤에 배치
- 서버를 추가하는 것만으로 수평 확장 가능
- 대규모 트래픽 상황에서도 대응 가능

즉, 사용자가 늘어나면 서버 수를 늘려 시스템을 확장가능


#### 3) 지연 시간(Latency)

읽기와 쓰기 요청 모두 매우 빠르게 처리되어야 한다.

- 쓰기 요청

    쓰기 요청은 복잡한 계산이나 과도한 데이터베이스 조회 없이 빠르게 처리할 수 있다.

    추가 최적화 예시

    - 카운터 값을 미리 생성
    - 단축 URL 후보를 메모리에 미리 저장

    이런 방식으로 URL 생성 속도를 더 높일 수 있다.

- 읽기 요청

    읽기 요청은 Redis 같은 **인메모리 데이터베이스**를 사용하므로 빠르게 처리할 수 있다.

    추가 최적화 예시

    - **LRU 캐시** 적용
    - 자주 조회되는 URL을 메모리에 유지

    즉, 단축 URL → 원본 URL 조회 속도를 매우 낮은 지연 시간으로 유지할 수 있다.


#### 4) 일관성(Consistency)

단축 URL 서비스는 설계상 **하나의 단축 URL당 하나의 데이터베이스 항목만 저장**한다.


매핑이 1:1로 유지되므로 높은 일관성을 확보할 수 있습니다.


#### 5) 내구성(Durability)

한 번 생성한 단축 URL 데이터는 절대 유실되지 않아야 한다.

이 설계에서는 **Redis** 를 사용하여 이 요구 사항을 충족한다.


#### 6) 신뢰성(Reliability)

단축 URL 서비스는 다음 상황에서도 정상적으로 동작해야 한다.

- 서버 장애
- 요청 폭주
- 일시적 시스템 중단
- 일부 구성 요소 장애

이 장에서 설계한 구조는 이러한 상황에서도 기능적 요구 사항을 만족할 수 있도록 구성되었다.


### 요구 사항 검토 요약

| 구분 | 요구 사항 | 충족 방식 |
|------|------|------|
| 기능 | 긴 URL → 단축 URL 생성 | URL 생성 API |
| 기능 | 단축 URL → 원본 URL 조회 | 조회 API |
| 기능 | 단축 URL 고유성 보장 | 고유 키/카운터 기반 생성 |
| 기능 | 단축 URL 길이 일정 | 7자리 고정 |
| 비기능 | 가용성 | 수평 확장 + SPOF 제거 |
| 비기능 | 확장성 | 로드 밸런서 + 서버 추가 확장 |
| 비기능 | 낮은 지연 시간 | Redis + 캐시 + 단순한 쓰기 처리 |
| 비기능 | 일관성 | 1 URL : 1 매핑 구조 |
| 비기능 | 내구성 | 영속 저장소 사용 |
| 비기능 | 신뢰성 | 장애 상황에서도 서비스 지속 가능 |

# 9.6 요약

이 장에서는 **URL 단축 서비스** 를 설계해 보았다.

먼저 이 서비스가 왜 유용한지, 실제로 어떤 상황에서 활용할 수 있는지 살펴보며 시스템 설계의 필요성과 중요성을 이해하고 이를 통해 URL 단축 서비스가 어느 정도 규모로 사용될 수 있는지도 파악 가능했다.

---

### 1. 요구 사항 정리

서비스 설계를 시작하기 전에 **기능적 요구 사항**과 **비기능적 요구 사항**을 정리한다.

기능적 요구 사항은 다음과 같이 나눌 수 있다.

- 핵심 기능
- 확장 기능

또한 서비스가 정상적으로 작동하기 위해 어떤 API가 필요한지도 함께 살펴봄


### 2. 규모 산정(Scale Estimation)

이후 시스템이 얼마나 큰 규모를 처리해야 하는지 계산

예를 들어

- 약 **3.5조 개 URL 처리**
- 필요한 저장 용량 계산
- 단축 URL의 문자 수 계산

이 과정을 통해 서비스의 **스케일**을 이해함


### 3. 단축 URL 생성 방식 검토

긴 URL에 고유한 단축 URL을 부여하기 위해 여러 가지 방법 검토

이 과정에서

- 각 방식의 장단점
- 발생 가능한 문제점
- 이를 해결하는 방법

을 함께 살펴보았고, 최종적으로 서비스 요구 사항을 가장 안정적으로 충족할 수 있는 방식을 선택했습니다.


### 4. 전체 아키텍처 설계

서비스 전체 구조를 아키텍처 다이어그램으로 정리하고, 데이터 흐름과 요청 처리 흐름을 살펴봄

예를 들어

- URL 생성 흐름
- URL 조회 흐름
- 캐시 사용 방식
- 카운터 서버/읽기 서버 역할

등을 구조적으로 정리


### 5. 요구 사항 최종 점검

마지막으로 설계한 시스템이 모든 요구 사항을 만족하는지 다시 확인했습니다.
