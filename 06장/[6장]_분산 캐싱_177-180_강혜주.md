# 배경

- 현대 컴퓨팅 환경이 빠르게 변화하며 확장성, 높은 성능을 갖춘 시스템 수요 증가
- 애플리케이션이 복잡해지고 사용자 수가 늘어나며 기존 데이터 검색 및 저장 방식의 한계

> 이러한 성능 문제를 해결하는 방법 중 하나가 바로 **캐싱(caching)**

# 개념

캐싱은 애플리케이션이나 사용자와 가까운 위치에 데이터를 임시로 저장하여 보다 빠르게 접근할 수 있도록 하는 기술이다.  
기본적으로 자주 사용하는 데이터는 고속 접근이 가능한 저장소 계층에 보관한다.

이 저장소는 보통 (동일 노드의)RAM, (동일 노드의)디스크, 네트워크 등 다양한 위치에 존재할 수 있으며, 이것으로 데이터 검색을 할 때 지연 시간을 줄이고 애플리케이션 성능과 사용자 경험을 크게 개선할 수 있다.

애플리케이션이 커지고 여러 서버나 데이터 센터에 분산되면 단일 캐시만으로는 충분하지 않을 수 있다.  
이때 **분산 캐싱**이 필요하다.

분산 캐싱은 마치 분산된 여러 도서관이 하나의 대형 도서관처럼 협력하는 방식과 비슷하다.  
각 도서관, 즉 노드는 자신만의 자료를 가지고 있지만, 네트워크로 서로 연결되어 있어 필요할 때 함께 동작하며 사용자가 원하는 책, 즉 데이터를 빠르게 찾을 수 있도록 돕는다.

이번 장에서는 캐싱 원리를 살펴보고, 더 나아가 분산 캐싱이라는 개념도 알아본다.

# 6.1 캐싱 정의

- 캐싱은 자주 사용하거나 계산 비용이 많이 드는 데이터를 더 빠르게 접근할 수 있는 위치에 저장하고 관리하는 기술을 말한다.
- 캐싱의 주된 목적은 데이터를 가져오는 데 필요한 시간과 리소스를 줄이는 데 있다.
    - 이를 위해 데이터를 복사하여 저장하는데, 이 복사본은 원본 데이터가 저장된 위치보다 훨씬 빠르게 접근할 수 있는 곳에 저장한다.


## 캐싱의 주요 개념

- **캐시**
    - 캐시는 자주 접근하는 데이터나 리소스의 복사본을 임시로 저장하는 공간이다.
    - 하드웨어 캐시(CPU 캐시), 소프트웨어 기반 캐시(인메모리) 등 여러 형태가 있다.

- **캐싱된 데이터(cached data)**
    - 캐시에 저장된 데이터의 복사본을 의미한다.
    - 이 데이터는 보통 데이터베이스나 디스크처럼 속도가 느리지만 영구적으로 저장되는 위치에서 가져오며, 빠르게 조회하려고 캐시에 보관한다.

- **캐시 히트(cache hit)**
    - 요청한 데이터가 캐시에 있을 때를 의미한다.
    - 이 경우 원본 데이터 소스에 접근할 필요 없이 캐시에서 데이터를 바로 가져와 빠르게 처리할 수 있다.

- **캐시 미스(cache miss)**
    - 요청한 데이터가 캐시에 없을 때를 의미한다.
    - 이 경우 시스템은 원본 데이터 소스에서 데이터를 가져오고, 이후 빠르게 조회할 수 있도록 해당 데이터를 캐시에 저장한다.

- **캐시 제거(eviction)**
    - 저장소에 캐시가 가득 차면 새로운 데이터를 저장하기 위해 기존 데이터를 삭제해야 하는 경우가 발생한다.
    - 이 과정에서 시스템은 적은 빈도로 사용하거나 최근에 사용하지 않은 데이터를 우선 제거하여 공간을 확보한다.

캐싱은 시스템 성능을 높이고 지연 시간을 줄여 사용자 경험을 개선하는 데 중요한 역할을 한다.

# 6.1.1 분산 캐싱 정의

- 분산 캐싱은 데이터에 더 빠르게 접근할 수 있도록 여러 서버나 노드에 자주 사용하는 정보를 메모리에 분산 저장하는 기술이다.
- 반복적으로 동일한 데이터를 원본 데이터 저장소에서 가져오는 대신, 분산 캐시는 이 데이터를 캐시에 미리 저장해 두어 시스템 응답 속도를 높이고 지연 시간을 크게 줄일 수 있다.

- 분산 캐싱의 핵심 목표는 디스크 기반 저장 시스템에서 데이터를 읽을 때 걸리는 속도 문제를 해결하는 데 있다.
- 여러 노드의 메모리에 캐시를 저장해 두면 디스크 입출력 없이, 또는 지연 없이 자주 사용하는 데이터를 빠르게 가져올 수 있다.
- 이 방식은 웹 애플리케이션, 데이터베이스, 대규모 데이터를 처리해야 하는 시스템 등에서 빠르게 데이터에 접근해야 할 때 특히 효과적이다.

---

# 6.1.2 분산 캐싱과 일반 캐싱의 차이점

분산 캐싱과 일반 캐싱은 모두 자주 사용하는 데이터를 저장하고 이를 빠르게 가져옴으로써 시스템 성능을 향상시킨다는 공통점이 있다.

하지만 이 둘은 적용 범위와 아키텍처, 운영 규모에서 차이가 있다.

| 적용 범위         | 일반 캐싱                                                                                                      | 분산 캐싱                                                                                                                     |
| ------------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **적용 범위**     | 일반적으로 캐싱은 자주 사용하는 데이터를 단일 서버나 로컬 캐시에 저장하고 가져온다. 이 캐시는 애플리케이션이나 시스템의 일부로 단일 기기나 서버에 있다.                     | 분산 캐싱은 캐싱 개념을 네트워크로 연결된 여러 노드나 서버로 확장한 방식이다. 분산 캐싱 시스템에서는 캐시가 여러 기기에 분산되어 있으며, 이 노드 간에 캐싱된 데이터를 공유할 수 있다.                 |
| **아키텍처**      | 데이터가 로컬 캐시에 저장되어 이 캐시는 보통 애플리케이션이 위치한, 애플리케이션과 동일한 기기 내에서 실행된다. 때문에 해당 캐시에 바로 접근할 수 있다.                    | 분산 캐시 노드는 구성된 네트워크를 기반으로 작동한다. 각 노드는 로컬 캐시를 가질 수 있으며, 노드 간 데이터를 공유하고 교환한다. 이런 아키텍처는 노드를 추가하여 수평적으로 확장할 수 있도록 설계되었다.       |
| **확장성**       | 단일 서버나 단일 애플리케이션 같은 소규모 환경에 적합하다. 로컬 환경에서 개선됨으로써 충분한 성능 향상을 얻을 수 있을 때 효과적으로 작동한다.                          | 대규모 시스템과 애플리케이션에서 발생하는 성능 문제를 해결하려고 설계되었다. 특히 여러 노드에서 데이터를 동시에 공유하고 빠르게 접근해야 하는 상황에서 성능과 확장성을 크게 높이는 데 유리하다.              |
| **활용 사례**     | 주로 서버나 애플리케이션에서 성능을 개선하고, 데이터베이스 접근 시간을 줄이며, 자주 사용하는 데이터를 빠르게 불러오는 데 활용된다.                                 | 분산 캐싱은 대규모 데이터 접근이 필요하거나 여러 시스템이 데이터를 공유해야 하는 환경에서 효과적이다. 대규모 웹 서비스, 분산 데이터베이스, 마이크로서비스 구조에서 자주 사용된다.                     |
| **일관성 및 동기화** | 일반 캐싱을 사용할 때는 캐시와 원본 데이터 간 일관성을 유지하는 작업이 비교적 단순하다. 하지만 분산 환경에서처럼 여러 노드를 동기화하거나 데이터를 동기화하는 과정이 훨씬 복잡하지는 않다. | 분산 캐싱 시스템은 각 노드가 동일한 최신 데이터를 공유할 수 있도록 설계된다. 이를 위해 노드 간 데이터 동기화, 프로토콜 기반 캐시 갱신, 일관성 관리 같은 전략을 이용하여 데이터 불일치 문제를 최소화하려고 한다. |

일반 캐싱과 분산 캐싱 모두 자주 사용하는 데이터를 저장하여 시스템 성능을 향상시키는 데 목적이 있다.

하지만 분산 캐싱은 이런 개념을 네트워크로 연결된 여러 노드로 확장하여 `대규모 분산 컴퓨팅 환경`에서 발생하는 문제를 해결한다.

일반 캐싱과 분산 캐싱 중 어떤 방식을 선택할지는 애플리케이션의 개발 요구사항과 규모에 따라 달라진다.

# 6.1.3 활용 사례

분산 캐싱을 사용하는 몇 가지 대표 사례를 살펴보자.

- **웹 애플리케이션**
    - 분산 캐싱은 사용자 세션, 페이지 구성 요소, 데이터베이스 쿼리 결과처럼 자주 접근하는 데이터를 저장하는 데 사용한다.

- **데이터베이스 쿼리 결과**
    - 자주 실행하는 데이터베이스 쿼리나 쿼리 결과를 캐싱하면 반복적으로 데이터베이스를 조회하는 것을 줄일 수 있어 효율성이 올라간다.

- **콘텐츠 전송 네트워크**
    - 콘텐츠 전송 네트워크는 에지 서버를 활용하여 이미지, 동영상, 스타일 시트 등 정적 콘텐츠를 저장하고 제공한다.

- **세션 관리**
    - 분산 캐시에 세션 데이터를 저장하면 웹 애플리케이션에서 세션 관리를 효율적으로 처리하고 확장성을 높일 수 있다.

- **API 응답 캐싱**
    - API 응답을 캐싱하면 백엔드 서버의 부하를 줄이고, 자주 요청하는 데이터를 더 빠르게 응답할 수 있다.

- **실시간 분석**
    - 자주 조회하거나 집계한 분석 데이터를 캐싱하면 실시간 보고서를 생성하고 대시보드를 빠르게 업데이트할 수 있다.

- **메시지 큐**
    - 메시지 큐를 캐싱하면 중간 결과를 저장하여 메시지 처리 시스템의 효율성을 높일 수 있다.

# 6.1.4 분산 캐싱을 사용할 때 장점

분산 캐싱을 사용하면 다음과 같은 장점이 있다.

- **성능 개선**
    - 자주 사용하는 데이터를 메모리에 저장하여 데이터 접근 속도를 크게 높인다.
    - 그 결과 시스템 응답 시간이 빨라지고 전반적인 성능이 향상된다.

- **확장 용이성**
    - 시스템 부하가 증가하면 캐시 노드를 추가해서 쉽게 확장할 수 있다.
    - 이렇게 하면 늘어나는 요청을 효율적으로 처리하여 성능을 안정적으로 유지할 수 있다.

- **백엔드 시스템의 부하 감소**
    - 자주 조회하는 데이터를 캐싱함으로써 분산 캐싱은 데이터베이스나 파일 시스템 등 백엔드 저장소에 가해지는 부담을 완화한다.
    - 이로써 리소스를 더욱 효율적으로 활용할 수 있으며, 백엔드로 전달되는 요청을 줄일 수 있다.
    - 특히 DDoS 공격 같은 상황에서는 원본 서버가 과부하로 다운되는 위험을 효과적으로 줄일 수 있다.

- **장애 허용성**
    - 분산 캐싱 시스템은 데이터 복제와 장애 허용성으로 특정 노드에 장애가 발생해도 다른 노드가 캐시 데이터를 처리할 수 있어 시스템의 안정성과 연속성을 유지한다.

- **일관된 접근 속도**
    - 캐싱은 자주 요청하는 데이터에 대해 데이터셋의 크기나 복잡성과 관계없이 일정한 속도로 접근할 수 있도록 한다.

- **비용 절감**
    - 성능이 향상되어 백엔드 시스템의 부하가 감소하면 추가 서버나 리소스가 필요하지 않아 인프라 비용을 줄일 수 있다.

- **사용자 경험 향상**
    - 응답 속도와 시스템 성능이 개선되면서 사용자에게 빠르고 자연스러운 경험을 제공한다.
    - 이는 사용자 만족도가 중요한 애플리케이션과 서비스에서 특히 중요한 요소다.

> 분산 캐싱은 다양한 컴퓨팅 환경에서 성능 저하 문제를 해결하고, 확장성과 보안을 강화하며, 리소스 활용을 최적화하는 역할을 한다.<br>
> 다만 어떤 장점이 두드러지게 작용할지는 애플리케이션 요구사항에 따라 다를 수 있다.
