### 어떤 방법을 선택해야 할까?

애플리케이션 서버와 함께 배포된 캐시와 독립적으로 배포된 캐시 중 하나를 선택하려면 애플리케이션의 요구 사항, 아키텍처, 우선순위를 종합적으로 고려해야 한다.

| 고려 요소       | 서버 통합형 캐시                                    | 스탠드얼론(Standalone) 캐시                       |
| ----------- | -------------------------------------------- | ------------------------------------------ |
| 성능 요구 사항    | 네트워크 오버헤드 없이 직접 접근 가능하여 빠른 응답 속도가 필요한 환경에 적합 | 높은 처리량과 대규모 트래픽 환경에 적합하며 독립적인 성능 확장 가능     |
| 확장성         | 애플리케이션과 함께 확장하는 구조로 비교적 단순한 확장 환경에 적합        | 캐시를 독립적으로 확장 가능하여 대규모 시스템에 유리              |
| 자원 공유       | 애플리케이션 서버와 메모리·CPU 자원을 함께 사용하여 효율적인 운영 가능    | 애플리케이션 서버와 분리되어 자원 충돌 없이 안정적인 운영 가능        |
| 유연성         | 설정과 통합은 간단하지만 기술 선택과 구성에는 제약 존재              | 요구 사항에 맞춰 다양한 캐싱 기술과 구성을 자유롭게 선택 가능        |
| 의존성 및 독립성   | 애플리케이션과 밀접하게 연결되어 있어 변경이나 장애가 서로 영향을 줄 수 있음  | 독립적으로 동작하여 특정 구성 요소 변경이 전체 시스템에 미치는 영향이 적음 |
| 운영 복잡성      | 배포와 관리가 비교적 단순하지만 세부 요구 사항 대응에는 한계 존재        | 설정과 관리가 더 복잡하지만 독립적인 수정·업데이트 가능            |
| 인프라 및 비용    | 서버 자원을 함께 사용하여 인프라 비용 절감 가능                  | 별도의 캐시 서버가 필요하여 추가 인프라 비용 발생 가능            |
| 네트워크 지연 허용성 | 네트워크 지연 시간을 최소화해야 하는 애플리케이션에 적합              | 캐시 접근 시 일정 수준의 네트워크 지연을 허용할 수 있는 환경에 적합    |

성능, 확장성, 유연성, 운영 효율성 등을 종합적으로 고려해서 신중히 선택하는 것이 중요!

### 요구 사항에 맞게 설계되었는지 점검하기

점검시 기능적·비기능적 요구 사항을 제대로 충족했는지 확인해 보면 좋다 

기능적 요구 사항은 put과 get 함수가 제대로 작동하니 만족

비기능적 요구 사항
  - **높은 성능**: 모든 연산이 ○(1)로 처리되기 때문에 매우 효율적
  - **높은 확장성**: 캐시를 배포하는 서버 수를 늘리고 이를 적절히 샤딩하면, 어떤 요청량이나 동시성에도 대응할 수 있어 확장성 측면에서 적합
  - **높은 가용성**: 이 부분은 여전히 개선이 필요합니다. 한 서버가 다운되면 해당 서버에 저장된 데이터 전체를 잃을 위험이 있다.

어떻게 해야 높은 가용성까지 챙길 수 있을까? -> 데이터 복제를 활용를 활용하면 가능하다!

동일한 데이터를 여러 서버에 복제하고 복사본을 다른 서버에 분산 저장함으로써 한 서버에 장애가 발생해도 다른 서버에서 요청을 처리할 수 있다.

보통 보조 복사본을 두 개 추가하여 총 세 개를 서로 다른 서버에 저장하고 이 중 하나는 주, 다른 두개는 보조로 활용한다.

보조 복사본은 읽기 요청이 많은 상황에서 부하를 분산하는 역할도 하며 쓰기 작업은 먼저 주 복사본에 저장한 후 이를 보조 복사본으로 복제한다.

하지만 데이터 복제를 활용하면 일관성의 문제가 있을 수 있다. 이 상황에선 어떤걸 우선할지 선택해야 한다.

## 6.3 대표적인 분산 캐시 솔루션

가장 많이 사용하는 분산 캐시 솔루션으로 레디스(Redis)와 맴캐시드(Memcached)가 있다.

각 솔루션의 특징을 알아보자

### 6.3.1 레디스

레디스는 문자열, 리스트, 집합, 해시 등 다양한 데이터 구조를 지원하는 인 메모리 데이터 저장소

단순한 저장소 기능을 넘어 데이터를 영구적으로 유지할 수 있는 옵션과 함께 발행·구독 메시징, 트랜잭션, Lua 스크립팅 등 고급 기능도 제공

**활용 사례**
  - 웹 애플리케이션 캐싱: 자주 요청하는 데이터를 캐시에 저장해서 빠르게 제공
  - 실시간 분석: 데이터를 빠르게 처리하고 분석해야 하는 환경에서 활용합
  - 세션 저장소: 사용자 세션 데이터를 효율적으로 관리
  - 리더보드 및 카운팅 시스템: 순위 계산이나 실시간으로 데이터를 집계하는 솔루션

**확장성**
  레디스는 데이터를 여러 노드에 나누어 저장하는 샤딩 방식을 이용하여 쉽게 확장 가능

**커뮤니티 지원**
- 활발한 오픈 소스 커뮤니티와 폭넓은 사용자 기반을 자랑
- 탄탄한 문서화와 커뮤니티 지원 덕분에 학습과 문제 해결이 수월

### 6.3.2 맴캐시드

캐시드는 단순하고 직관적인 키-값 저장소로 메모리 기반 캐싱 솔루션

가볍고 사용하 기 쉬운 설계로 기본적인 데이터 타입을 효율적으로 지원하며, 분산 아키텍처로 확장성도 제공한다.

**활용 사례**
- 웹 애플리케이션 캐싱: 자주 사용하는 데이터를 캐시에 저장해서 빠르게 처리
- 세션 저장소: 사용자 세션 데이터를 효율적으로 관리
- 데이터베이스 결과 캐싱: 데이터베이스 쿼리 결과를 캐시에 저장해서 반복 요청 처리 속도가 향상
- 단순 키-값 분산 시스템: 단순한 데이터 저장이 필요한 분산 시스템에서 효과적으로 사용한다.

**확장성**
- 맴캐시드는 캐시 클러스터에 노드를 추가하여 수평 확장이 가능
- 일관된 해싱 방식을 이용하여 데이터를 여러 노드에 분산 저장

**커뮤니티 지원**
- 신뢰할 수 있는 안정적인 코드를 기반으로 하기에 널리 사용
- 사용법이 간단하고 다양한 프로그래밍 언어와 쉽게 통합가능

### 6.3.3 레디스와 맴캐시드 중 어떤 것을 선택해야 할까?

| 비교 항목     | Redis                                | Memcached                        |
| --------- | ------------------------------------ | -------------------------------- |
| 사용 목적     | 다양한 데이터 구조를 지원하여 활용 범위가 넓음           | 단순한 키-값 캐싱에 최적화되어 빠르고 간단하게 사용 가능 |
| 데이터 보존 여부 | 영구 저장(Persistence) 옵션 지원으로 데이터 보존 가능 | 메모리 기반 캐시로 기본적으로 데이터 영구 저장 불가능   |
| 데이터 구조 지원 | 문자열, 리스트, 셋, 해시 등 다양한 데이터 구조 지원      | 단순한 키-값(Key-Value) 구조 중심         |
| 유연성       | 복잡한 데이터 처리와 다양한 기능 구현에 유리            | 단순 캐싱 환경에서 효율적                   |
| 사용 편의성    | 기능이 다양해 학습과 설정에 시간이 필요할 수 있음         | 구조가 단순하여 쉽게 통합하고 운영 가능           |

레디스와 맴캐시드 중 무엇을 선택할지는 애플리케이션의 요구 사항, 데이터 처리의 복잡성, 사용 편의성, 커뮤니티 지원 같은 요소를 어떻게 우선시하느냐에 달려 있다.

### 요약

이 장에서는 캐싱과 분산 캐싱의 개념 및 특징을 학습했고, 캐싱은 자주 사용하는 데이터를 빠르게 조회하기 위한 기술이며, 분산 캐싱은 데이터를 여러 서버에 분산 저장해 성능과 확장성을 높이는 방식

또한 분산 캐싱의 장점인 성능 향상, 확장성, 부하 감소와 함께 데이터 일관성, 캐시 무효화, 운영 복잡성 같은 한계를 알아보고 마지막으로 분산 캐시 설계 과정과 대표 솔루션인 Redis, Memcached 의 특징과 선택 기준을 정리하였다.
