## 4.3 로드밸런서

로드 밸런싱은 여러 서버, CPU, 하드 드라이브, 네트워크 연결에 작업을 고르게 분배하여 자원을 효율적
으로 사용하고, 처리 속도를 높이고, 응답 시간을 줄이고, 시스템이 과부하에 걸리지 않도록 한다.


![그림4-7.png](image/%EA%B7%B8%EB%A6%BC4-7.png)

**로드 밸런싱의 주요 특징**

- 트래픽을 여러 서버에 고르게 분산시켜 특정 서버에 과부하 방지
- 장애가 발생하거나 속도가 느려진 서버의 요청을 다른 정상 서버로 재전송하여 시스템 내구성을 높임
- 일부 서버가 작동하지 않더라도 나머지 서버가 요청을 처리할 수 있어 서비스 가용성 유지 가능
- 높은 트래픽 상황에서도 시스템이 완전히 멈추지 않고 성능이 점진적으로 저하되도록 하여 부하가 늘어나면 응답시간이 길어질수 있지만 계속 운영 가능

로드 밸런싱은 하드웨어 장치나 소프트웨어를 사용하여 트래픽을 분산하는 방식이다.

### 4.3.1 로드 밸런서 위치

![그림4-8.png](image/%EA%B7%B8%EB%A6%BC4-8.png)

클라이언트와 서버 사이에 로드 밸런서를 배치하면 들어오는 클라이언트 요청을 여러 서버 인스턴스로 분산할 수 있다<br>
또 여러 계층으로 된 구조의 각 단계 사이에도 로드 밸런서 배치가 가능하다

시스템의 여러 지점에서 트래픽을 분산시키면 각 계층을 독립적으로 보호할 수 있다. 이는 특정 계층에서 문제를 감지하면 트래픽을 우회시켜 전체 시스템이 원활하게 운영되도록 돕는 역할을 한다.

### 4.3.2 로드 밸런서의 장점

- 헬스 체크'를 이용하여 서버가 정상 작동 중인지 파악할 수 있다. 이것으로 클라이언트가 접근할 수 없는 서버에 요청을 보내지 않도록 막아 준다.

- TLS(전송 계층 보안)연결을 종료하여 서버의 안정성을 높일수 있다. -> 공격자가 TLS를 이용하여 서버에 과부하를 주는 것을 방지할수 있다.<br>
로드 밸런서가 TLS연결을 종료할때는 트래픽에 대한 보안 게이트웨이 역할을 한다. -> 암호화된 데이터를 백엔드로 직접 보내는 대신 로드밸런서가 데이터를 복호화 하여 서버는 복호화에 자원을 쓰지않아도 되므로 본연의 작업에 집중할 수있다.
- 트래픽 패턴을 분석하고 효율적인 요청분배 알고리즘을 통해 서버 리소스 활용을 극대화한다.

### 4.3.3 전역 로드 밸런싱과 로컬 로드 밸런싱

전역 서버 로드 밸런싱(GSLB)은 다양한 지역에 있는 데이터 센터 간에 트래픽을 분산시키는 역할을 하고, 로컬 로드 밸런싱
은 개별 데이터 센터 내에서 자원 활용도를 높인다.

![그림4-9.png](image/%EA%B7%B8%EB%A6%BC4-9.png)

전역 서버 로드 밸런싱은 사용자 위치, 데이터 센터 상태, 서버 수 같은 요인을 기반으로 전역 트래픽을 적절한 데이터 센터로 전달한다.<br>
전력이 끊기거나 네트워크 장애가 발생했을 때도 자동으로 다른 데이터센터로 전환하는 기능과 온프레미스(on-premises) 방식이나 서비스 형태(로드 밸런싱 서비스(LBaaS))로도 사용할 수 있다

각 데이터 센터에는 해당 지역 내 서버로 트래픽을 분산하는 로컬 로드 밸런싱이 배치되고, 로컬 로드 밸런싱은 전역 서버 로드 밸런싱에 서버 상태와 용량 정보를 전달하며,
전역 서버 로드 밸런싱은 이 데이터를 활용하여 들어오는 트래픽을 어떤 데이터 센터로 보낼지 결정한다.
