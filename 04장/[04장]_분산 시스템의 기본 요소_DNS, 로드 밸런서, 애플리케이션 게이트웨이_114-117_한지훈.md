### DNS와 전역 로드 밸런서

- DNS도 전역 로드 밸런싱 역할을 어느정도 수행할 수 있다. 하나의 DNS 쿼리에 여러 IP 주소를 반환하고 각 클라이언트가 순차적으로 다른 IP를 받도록 해 라운드 로빈 방식으로 트래픽을 여러 데이터 센터로 분산시킨다.


- 하지만 라운드 로빈 방식의 DNS 로드 밸런싱에는 한계가 있다. ISP별로 발생하는 트래픽 차이를 반영하지 못하고 서버가 다운되었을 때 이를 감지할 수 없다. 그래서 DNS는 로드 밸런싱 효과를 높이기 위해 캐시의 TTL을 짧게 설정하는 방식을 사용한다.

>라운드 로빈 방식
> 
> 라운드 로빈 방식은 요청을 순차적으로 분배하는 부하 분산이다. 예시로 A,B,C가 있다 할 때 첫번째 요청은 서버 A 두번쨰 요청은 B, 세번째는 C로 보내고 다음 요청은 다시 A로 돌아가서 반복하는 방식이다. 간단하면서도 트래픽을 고르게 분산하는 데 유용하지만 각 서버의 상태나 처리 능력은 고려하지 않아 균등하게 부하 분산하기 어려울 수 있다.


 - 전반적으로 저녁 로드 밸런서 및 로컬 로드 밸런싱 계층은 여러 데이터 센터와 서버로 구성된 시스템으로 들어오는 트래픽을 효율적으로 분산하려고 함께 작동한다.
 - DNS 로드 밸런싱은 전역 트랙=픽을 어느 정도 분산할 수 있지만 한계가 있어 각 데이터 센터 내부에 로컬 로드 밸런서가 필요하다는 한계점을 지닌다. DNS 로드 밸런싱만으로 트래픽 분산에 한계가 생기는 이유는 다음과 같다.
   - DNS 패킷 크기가 512 바이트로 작아 모든 서버 IP 주소를 응답에 포함할 수 없다. 이 떄문에 로드 밸런싱 기능이 제한된다.
   - 클라이언트가 DNS가 반환한 여러 IP 주소 중 하나를 임의로 선택하기에 과부하가 걸린 데이터 센터로 연결될 수 있다. 이 경우 DNS는 클라이언트가 어떤 IP 주소를 선택할지 간섭할 수 없다.
   - DNS는 특정 클라이언트와 가장 가까운 서버를 결정하지 못하지만 지리적 위치 기반 연결이나 애니캐스팅 같은 기술을 활용하면 도움될 수 있다. 하지만 이런 기술은 적용이 간단하지 않다.
   - 장애가 발생시 특히 TTL 값이 길 때 캐싱으로 DNS를 이용한 복구가 느릴 수 있다.

 - 이 문제점은 각 데이터 센터 내부에 로컬 로드 밸러서를 두어야 막을 수 있다.
 - 클라이언트 애플리케이션은 가상 IP 주소를 사용해 로컬 로드 밸런서에 연결하고 로컬 로드 밸런서는 클라이언트 요청을 데이터 센터 내 현재 정상적으로 동작하고 있는 서버에 효율적으로 분배한다.
 - 요약하자면 DNS 로드 밸런싱은 전 세계 데이터 센터 간 트래픽을 기본적으로 분산하는 역할을 한다. 그러나 각 데이터 센터 내부에서는 로컬 로그 밸런서가 더욱 효과적이고 정교한 방식으로 트래픽을 분배해 준다. 로컬 로드 밸런서는 백엔드 서버 간 부하를 최적으로 분산할 수 있는 기법을 사용할 수 있다. 결국 데이터 센터와 글로벌 차원 모두에서 높은 성능과 확정성을 달성하려면 전역 및 로컬 로드 밸런싱이 모두 필요하다.


### 4.3.5 로드 밸런서가 사용하는 알고리즘


 - 로드 밸런서는 다양한 알고리즘을 사용해 들어오는 클라이언트 요청을 백엔드 서버에 어떻게 분배할지 결졍하는데 일반적으로는 다음과 같은 알고리즘을 사용한다.
   - 라운드 로빈 스케줄링 : 각 요청을 순차적으로 다음 서버에 할당하는 방식이다. 서버 목록을 순회하면서 요청을 분배하므로 가장 간단한 알고리즘이지만 서버 간 성능 차이를 고려해 요청을 분배할 수 있다.
   - 가중치 기반 라운드 로빈 : 서버 용량에 따라 가중치를 부여하고 가중치가 높은 서버가 더 많은 요청을 받도록 설정한다. 이것으로 서버 간 성능 차이를 고려해 요청을 분배할 수 있다.
   - 최소 연결 알고리즘 : 현재 연결된 요청 수가 가장 적은 서버에 새로운 요청을 할당하는 방식이다. 특히 처리 시간이 긴 요청이 포함되었을 때도 서버 간 부하를 고르게 분배할 수 있도록 도와준다. 이를 위해 로드 밸런서는 각 서버의 연결 상태를 파악하고 있어야 한다.
   - 최소 응답 시간 알고리즘 : 응답 시간이 가장 짧은 서버에 요청을 할당해 성능이 중요한 서비스에서 효율을 극대화한다.
   - IP 해시, URL 해시, 일관성 해시 알고리즘 : 각각 클라이언트의 IP 주소나 요청 URL을 기반으로 요청을 특정 서버에 할당하는 방식이다. 이는 특정 클라이언트나 URL 요청을 특정 서버로 라우팅해야 하는 경우 유용하다.


 - 일반적으로 로드 밸런서는 자원 활용을 극대화하고 처리 속도와 응답 시간을 최적화해 장애에도 안정적으로 동작할 수 있도록 서버 용량, 부하, 요청 특성, 애플리케이션 요구 사항 등을 고려해서 다양한 알고리즘을 선택한다. 단순한 알고리즘을 사용하면 부하를 고르게 분한하는 데 한계가 있을 수 있고, 정교한 알고리즘을 사용하면 서버 상태를 지속적으로 추적하고 관리해야 하는 부담이 따를 수 있다. 따라서 상황에 맞는 알고리즘 선택이 중요하다.

### 정적 알고리즘 VS 동적 알고리즘


 - 로드 밸런싱 알고리즘은 서버 상태를 고려하는지에 따라 정적 알고리즘과 동적 알고리즘을 구분할 수 있다. 정적 알고리즘은 서버 상태를 실시간으로 반영하지 않고 미리 설정된 서버 구성에 따라 요청을 분배한다. 그래서 주로 간단한 로드 밸런싱에 사용한다.
 - 동적 알고리즘은 서버의 상태나 최근 상태를 고려해 요청을 분배한다. 이런 알고리즘은 서버와 통신해 상태 정보를 유지해야 하므로 통신 비용이 증가하고 시스템이 다소 복잡하다는 단점이 있다. 그러나 동적 알고리즘을 사용하면 로드 밸런서들이 정보를 교환해 결정을 내리기 때문에 시스템이 더욱 유연하게 작동한다. 또 활성 상태인 서버로만 요청을 전달한다.
 - 동적 알고리즘은 정적 알고리즘보다 더 복잡하지만 최신 서버 상태를 반영하여 결정을 내리므로 더 효과적으로 부하 분산을 할 수 있습니다. 정적 알고리즘ㅇ과 달리 시간에 따라 변화하는 상황에 우연하게 대응할 수 있다.
 - 실제로 동적 알고리즘을 더 많이 사용한다. 좀 더 복잡하더라도 부하 분산 효과가 뛰어나기 때문이다. 하지만 서버가 적은 단순한 환경에서는 정적 알고리즘을 사용하는 것만으로도 충분할 수 있다. 정적 알고리즘과 동적 알고리즘을 선택할 때는 단순성, 성능, 적응성 사이의 균형을 고려해야 한다.
 - 정리하자면 백엔드 서버의 부하 상태와 응답 속도 등을 정확히 파악하면 로드 밸런서가 자원을 최적으로 활용해 성능이 떨어지지 않도록 효과적으로 요청을 분배할 수 있다.

### 스테이트풀 VS 스테이트리스

 - 로드 밸런서는 클라이언트의 세션 정보를 유지하는지에 따라 스테이트풀과 스테이트리스로 분류할 수 있다.
 - 스테이트풀 로드 밸런서는 클라이언트와 백엔드 서버 간에 연결 상태 정보를 저장한다. 이 상태 정보를 로드 분배 알고리즘에 반영해 요청을 어느 서버로 보낼지 결정한다. 하지만 스테이트풀 로드 밸런서는 모든 로드 밸런서가 상태 정보를 공유해야 하므로 시스템이 복잡하고 확장성에 한계가 찾아오기 쉽다. 스테이트풀 로드 밸런서는 백엔드 서버에 연결된 모든 클라이언트의 세션 정보를 관리하는 구조를 갖추고 있다. 기존 세션에서 요청이 들어오면 저장된 세션 정보를 바탕으로 요청을 적절한 백엔드 서버로 전달할 수 있다.
 - 반면 스테이트리스 로드 밸런서는 클라이언트의 세션 상태를 따로 저장하지 않는 대신, 일관된 해싱 알고리즘을 사용해 요청을 서버에 매핑한다. 이 방식은 성능과 확장성 면에서 유리하지만 인프라가 변경될 때 안정성이 상대적으로 낮을 수 있다. 서버가 추가되거나 제거되거나 일시적으로 사용 불가능할 때는 일관된 해싱만으로는 기존 세션 요청을 항상 올바른 서버로 라우팅하기 어렵다. 따라서 이런 상황에서는 추가적인 로컬 상태 정보가 필요하다.
 - 일반적으로 여러 로그 밸런서에서 상태 정보를 공유하는 방식을 스테이트풀 로드 밸런싱이라고 한다. 단일 로드 밸런서 내에서만 로컬 상태를 유지하면서 요청을 처리하는 방식을 스테이트리스 로드 밸런싱이라 한다.
 - 스테이트풀과 스테이트리스 로드 밸런싱을 선택할 때는 안정성, 확장성, 복잡성 사이에서 균형을 잘 맞추어야 한다. 스테이트풀 방식은 가용성과 신뢰성을 높일 수 있는 반면, 스테이트리스 방식은 속도가 빠르고 확장성이 높고 구현이 더 간단하다. 실제 운영 시스템에서는 이 두가지 방식을 혼합해 사용할 떄가 많다.
 - 결론적으로 시스템의 요구 사항과 아키텍처에 따라 스테이트풀과 스테이트리스 로드 밸런싱 기술을 적절히 활용하는 것이 중요하다. 이제 로드 밸런서의 종류를 알아보자.

> 스테이트풀과 스테이트리스를 택배 회사 예시로 이해하기
> 
> 택배 회사가 많은 배송 요청을 처리해야 하는 상황을 생각해 보자.
> - 스테이트풀 : 이 택배 회사에서는 고객 택배를 처리할 때 담당 택배 기사와 배송 경로를 미리 지정해 두고 그 정보를 계속 유지한다. 즉, 특정 고객의 택배는 항상 같은 택배 기사가 같은 경로로 배송한다. 고객이 중간에 추가 배송을 요청하더라도 해당 기사와 경로가 이미 정해져 있어 바로 그 정보를 바탕으로 처리할 수 있다. 이 방식은 고객과 연속성을 보장하지만, 택배 기사 간 정보를 계속 공유하고 업데이트해야 해서 시스템이 복잡할 수 있다.
> - 스테이트리스 : 이 회사에서는 매번 택배를 새로 맡는 기사에게 배정합낟. 고객이 추가 배송을 요청하면 택배는 다시 무작위로 배정되기 때문에 이전의 기사와 경로 정보는 유지되지 않는다. 이 방식은 빠르고 효율적이지만 고객이 연속성을 기대하기는 어려울 수 있다. 예시로 택배 두 개가 다른 기사에게 배정되어 각각 다른 시간에 도착할 수도 있다.
> 
> 두 방식을 조합한 새로운 택배 시스템
> 
> 현실에서는 스테이트풀과 스테이트리스 방식을 적절히 조합해 효율성을 극대화할 수 있다. 예시로 VIP 고객이나 특별 관리가 필요한 고객이라면 스테이트풀 방식을 적용해 같은 기사가 지속적으로 담당하게 하고 일상적인 배송은 스테이트리스 방식으로 유연하게 배정할 수 있다.
> 
> 이 새로운 택배 시스템에서 VIP 고객은 매번 같은 기사에게 배정되어 더 세심한 관리가 가능하고 일반 고객은 매번 다른 기사에게 배정되어 더 세심한 관리가 가능하고 일반 고객은 매번 다른 기사에게 배정되어 빠르게 처리 가능하다. 이렇게 하면 고객 요구 사항에 맞는 연속성과 효율을 동시에 갖출 수 있어 택배 회사는 시스템 부담을 줄이면서 고객 기대를 충족시키는 유연한 서비스를 제공할 수 있다. 

### 4.3.6 OSI 모델의 각 계층에서 로드 밸런싱

 - 로드 밸런서는 OSI 모델의 여러 계층에서 작동하며 각 계층에 따라 기능과 성능이 달라질 수 있다. OSI 모델은 시스템의 내부 구조나 특정 기술과 관계없이 네트워크 기능을 표준화하는 개념적 프레임워크이다. 물리적 계층, 데이터 링크 계층, 네트워크 계층, 전송 계층, 세션 계층, 프레젠테이션 계층, 응용 계층으로 나눌 수 있다. 이런 계층 구조는 네트워크 프로토콜의 개발, 문제 해결, 관리 등을 효율적으로 다룰 수 있어 네트워크 아키텍처의 상호 운용성과 유연성을 높인다.
 - 전송 계층(4계층) 로드 밸런서는 TCP와 UDP 프로토콜을 기준으로 부하를 분산한다. 이 로드 밸런서는 TCP/UDP 연결을 유지하면서 클라이언트 요청을 백엔드의 동일한 서버로 전달하여 연결 지향 프로토콜에 안정성을 제공한다. 또 일부 로드 밸런서는 TLS 종료 기능도 지원한다. 반면에 응용 계층 로드 밸런서는 HTTP 등 응용 계층 데이터를 기반으로 부하를 분산한다. HTTP 헤더 URL, 쿠키, 사용자 ID 등 애플리케이션 정보를 활용해 더 정교한 요청 분배가 가능하다. TLS 종료 외에도 7계층 로드 밸런서는 속도 제한, HTTP 라우팅, 헤더 재작성 등 다양한 기능을 수행할 수 있다.
 - 응용 계층 로드 밸런서는 요청 내용을 기반으로 한 애플리케이션 특성을 더 많이 고려해 부하를 최적화 할 수 있는 반면 전송 계층 로드 밸런서는 더 낮은 계층에서 동작해 속도가 상대적으로 빠르다.
 - 실제 대부분의 로드 밸런싱 솔루션은 성능 요구사항, 애플리케이션 요구 사항, 지원하는 프로토콜에 맞추어서 전송 계층 방식과 응용 계층 방식을 함께 사용한다. 전송 계층 로드 밸런서는 TCP 기반 애플리케이션에 기본적인 부하 분산과 안정성을 제공하고 응용 계층 로드 밸런서는 요청의 URL, HTTP 헤더, 쿠키 등 애플리케이션 세부 정보를 바탕으로 더 정교하고 상황에 맞춘 부하 분산을 구현할 수 있다.
 - 전송 계층과 응용 계층 로드 밸런싱 중 어느 방식을 선택할지는 애플리케이션 아키텍처와 사용하는 프로토콜, 요구 사항, 부하 분산을 얼마나 세밀하게 제어하고 최적화할지에 따라 다르다. 두 방식 모두 높은 가용성, 확장성, 자원 활용의 극대화라는 목표를 달성하는 데 장점이 있다. 다음으로 로드 밸런서의 배치 방법을 살펴보자.
