### 재귀적 쿼리 동작방식

![그림4_4.png](image/%EA%B7%B8%EB%A6%BC4_4.png)

- 요악하면 반복적 쿼리에는 로컬 리졸버가 DNS 계층 구조를 직접 거쳐 가며 IP 주소를 찾고 반면에 재귀적 쿼리에서는 로컬 리졸버가 상위 서버에 순차적으로 요청을 보내고 이 서버들이 계층을 따라 쿼리를 전달해서 최종 IP 주소를 얻습니다.


### 반복적 쿼리는 로컬 리졸버가 아닌 내 컴퓨터가 직접 요청하는 것이 아닌가?

 - 위 설명에서 반복적 쿼리는 사용자 컴퓨터가 직접 여러 서버에 요청을 보내며 IP 주소를 찾는 방식이라고 이해할 수 있지만 실제로는 사용자 컴퓨터가 로컬 리졸버에 한 번 요청을 보내면 그 이후 과정을 로컬 리졸버가 담당한다.

### 반복적 쿼리와 재귀적 쿼리의 차이
 - 반복적 쿼리에서 로컬 리졸버는 각 DNS 서버(루트 서버, 최상위 도메인 서버, 권한 있는 네임 서버)에 하나씩 요청을 보내며 필요한 정보를 찾는다. 사용자는 로컬 리졸버에 한 번만 요청하고 로컬 리졸버가 모든 과정을 직접 처리하며 최종 IP 주소를 찾는다.
 - 반면에 재귀적 쿼리에서는 로컬 리졸버가 요청을 받은 후 상위 서버에 요청을 한 번만 보내고 이후 모든 과정은 상위 서버가 알아서 처리한다. 최종 IP 주소를 찾으면 해당 정보를 로컬 리졸버에 전달하고 다시 사용자는 최종 결과를 받아본다.

   - 반복적 쿼리: 사용자는 로컬 리졸버에게 요청을 보내고 로컬 리졸버는 루트 서버, 최상위 도메인 서버, 권한있는 네임 서버 등에게 요청에 대한 처리를 진행하여 찾는다. 로컬 리졸버가 다른 서버에게 요청해 찾는다.

![그림4_5.png](image/%EA%B7%B8%EB%A6%BC4_5.png)

   - 재귀적 쿼리: 사용자는 도서관의 로컬 리졸버에게 요청하고 로컬 리졸버는 루트 서버에게 요청을 전달하고 루트 서버는 요청을 최상위 도메인서버로 넘기고 최종적으로 권한 있는 네임 서버로 요청이 넘어간다. 마지막에 권한있는 네임 서버가 응답을 반환하게 되면 로컬 리졸버에게 요청이 반환된다.

![그림4_6.png](image/%EA%B7%B8%EB%A6%BC4_6.png)

- 웹 서비스를 개발할 때 프론트나 백엔드 서비스에서 서버 주소를 .env 파일 같은 환경 변수에 저장해 사용한다. 예시로 .env 파일에 SERVER_API = https://api.my-server.com 이런 방식이다.
- 이렇게 도메인 이름 대신 IP를 직접 넣는다면?
- SERVER_API = 192.168.1.1 처럼 IP 주소를 바로 사용하면 네트워크가 도메인 이름을 IP로 변환하는 DNS 쿼리 과정을 건너뛸 수 있어 응답 지연 시간을 줄이는데 유리할 수 있다. 하지만 현업에서 주로 도메인 이름 사용하는 이유는 다음과 같다.
  - 유연성: 도메인 이름을 사용하면 서버의 IP 주소가 변경되더라도 도메인을 통해 연결할 수 있어 코드나 환경변수를 바꾸지 않아도 된다. 반면에 IP 주소를 직접 사용하면 서버 IP가 바뀔 때마다 일일이 업데이트 해야한다.
  - 로드 밸런싱: 많은 서비스에서 여러 IP 주소로 트래픽을 분산하는 로드 밸런싱을 사용한다. 예시로 my-server.com 이라는 도메인에 연결된 IP 주소가 여러개 있다면 DNS는 접속 요청을 분산해 각기 다른 서버로 나누어 보낼 수 있다. 덕분에 특정 서버에 과부하가 걸리지 않은 채 더 빠르고 안정적으로 서비스를 제공할 수 있다. IP 주소를 고정하면 이런 로드 밸런싱이주는 혜택을 누리기 어렵다.
  - 보안: 도메인 이름을 사용해 SSL/TLS 인증서를 검증하는 과정이 필요할 때가 많다. 예시로 https://my-server.com에 접속할 때 웹 브라우저는 이 도메인의 인증서를 확인하여 안전한 서버임을 보장받는다. 그러나 IP 주소로만 접속하면 인증서 확인이 어렵고 보안 경고가 표시 될 수 있다. 도메인 이름을 사용하면 인증서를 통한 신뢰성 검증이 가능하므로 보안 측면에서 안전하다.
- 개발 단계에서는 테스트 목적으로 IP 주소를 직접 설정해도 괜찮을 수 있다. 하지만 운영 단계에서는 유연성, 로드밸런싱, 보안 문제를 도려하여 도메인 이름을 사용하는 편이 더 바람직 하다. 이렇게 하면 서비스를 안정적이고 확장 가능하게 운영할 수 있고 유지 보수 또한 수월하다.
- 결론적으로 지연 시간을 줄일 수 있다는 장점은 있지만 유연성, 로드 밸런싱ㅡ 보안 측면에서 도메인 이름을 사용하는 편이 더 적합하기에 현업에서는 주로 도메인 이름을 환경 변수애 사용한다.

> 인프라 부하를 줄이려면 어떤 메커니즘이 적합할까
> 
> 반복적 쿼리이다. 재귀적 방식은 DNS 요청을 처리할 때 단계마다 상태를 계속 유지해야 해서 메모리를 더 차지하고 시간도 지날수록 느려질 수 있다. 하지만 반복적 쿼리는 상태를 따로 관리하지 않아도 되므로 이런 문제에서 자유롭기 때문이다.


- 웹 사이트 이름을 입력하면 거의 즉시 해당 웹 페이지각 나타나는데 이렇게 빠른 인터페이스를 제공할 수 있는 것은 바로 DNS 캐싱 덕분이다.
