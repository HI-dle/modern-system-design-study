# 8.4 API 보안

API 보안은 시스템 설계에서 매우 중요한 요소다.

API는 소프트웨어의 구성 요소 간 연결 고리 역할을 하며, 공격자가 노리기 쉬운 대상이다. 때문에 강력한 보안 대책을 이해하고 적용하는 것이 아주 중요하다. 

# API 보안의 핵심인 인증(authentication)과 인가(authorization)

- 인증: “사용자가 누구인가?” - 사용자가 아이디와 비밀번호로 로그인하는 것과 같다. 
- 인가: “이 작업을 해도 되는가?” - 사용자가 특정 데이터나 작업에서 권한이 있는지 확인한다. 

## 8.4.1 인증

인증은 사용자, 시스템, 애플리케이션의 신원을 확인하는 과정으로 API 보안의 첫 번째 방어선이다.

시스템에 접근하는 사용자가 올바른지 확인하는 과정이다.

### 인증 구현 방법과 각각의 장단점

| 인증 방식         | 설명                                                                                                            | 장점                                                                                                          | 단점                               |
| ------------- |---------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------| -------------------------------- |
| **API Key**   | 클라이언트가 요청 헤더에 API 키를 포함하여 인증                                                                                  | 구현이 매우 간단함                                                                                                  | 키가 노출되면 누구나 사용 가능, 세밀한 권한 제어 어려움 |
| **OAuth 2.0** | 사용자가 비밀번호를 직접 제공하지 않고 제3자 서비스에 제한된 리소스 접근 권한을 위임하는 인증 방식                                                      | 높은 보안성, 권한 범위(Scope) 설정 가능, 다양한 환경 지원(OAuth 2.0은 웹, 데스 크톱, 모바일 애플리케이션, 스마트 기기 등 다양한 환경에 맞춘 인증 절차(flow)를 제공) | 구현이 복잡함                          |
| **JWT**       | JSON 형식의 정보를 서명하여 토큰으로 전달하는 방식. 정보는 JSON 형식으로 작성된 클레임(claim)으로 구성되며, 클레임은 JWS(JSON Web Signature)구조로 감싼다.(메시지 인증 코드(MAC)로 디지털 서명을 하거나 무결성을 확인할 수 있으며, 필요하면 암호화도 가능) | Stateless 구조, 크기가 작음, URL로 전송하기에 적합(시스템이나 네트워크 간 데이터 전송에 자주 사용)                                                                       | 토큰 탈취 시 만료 전까지 사용 가능, 토큰 폐기 어려움  |

## 옮긴이 노트: API 키의 의미

API 키는 일종의 '입장권' 처럼 특정 API에 접근하는 인증 수단이다. 

클라이언트는 이 키를 서버에 전송함으로써 자신이 요청을 보낼 자격이 있음을 증명한다.

입장권이 유출되면 누구나 행사장에 들어갈 수 있듯이, API 키가 노출되면 공격자가 이를 악용할 위험이 있다.

때문에 API 키는 간단한 인증 수단으로 유용하지만, 높은 보안 수준이 요구될 때는 더 강력한 인증 방식(OAuth 등)을 사용하면 좋다.

API 키는 일반적으로 HTTP 요청의 헤더에 포함하여 전달. 

### API 키를 활용한 검증 과정 코드로 살펴보기.

#### 서버 코드: API 키 검증 

서버에서는 클라이언트가 보낸 API 키가 유효한지 확인. 

- 서버 측에서 API 키를 검증하는 파이썬 예시 코드.

  ```python
  # 유효한 API 키 목록
  VALID_API_KEYS = {"12345", "67890", "abcde"}
  
  # API 키 검증 함수
  def check_api_key(api_key):
      if api_key in VALID_API_KEYS:
          print("Access granted. Welcome!")
          return True  # 접근 허용
      else:
          print("Access denied. Invalid API key!")
          return False  # 접근 거부
  
  # 예제 실행
  user_api_key = "12345"  # 클라이언트가 보낸 API 키
  check_api_key(user_api_key)  # 유효성 확인
  ```
#### 클라이언트 코드: 자바스크립트로 API 요청 보내기
클라이언트는 API 키를 HTTP 요청의 Authorization 헤더에 포함하여 서버에 보낸다. 

- fetch API를 사용한 요청 자바스크립트 예시 코드

  ```javascript
  // 서버의 엔드포인트 URL
  const API_URL = "http://example.com/api/resource";
  
  // 클라이언트가 사용하는 API 키
  const API_KEY = "12345"; // 유효한 API 키
  
  // HTTP 요청 옵션
  const options = {
    method: "GET", // 요청 방식
    headers: {
      Authorization: `Bearer ${API_KEY}`, // 헤더에 API 키 추가
      "Content-Type": "application/json", // 요청 데이터 형식
    },
  };
  
  // 서버에 요청 보내기
  fetch(API_URL, options)
    .then((response) => {
      if (!response.ok) {
        throw new Error(`HTTP error! status: ${response.status}`);
      }
      return response.json(); // JSON 응답 파싱
    })
    .then((data) => {
      console.log("응답 데이터:", data); // 응답 데이터 출력
    })
    .catch((error) => {
      console.error("오류 발생:", error);
    });
  ```

### 참고: Bearer 토큰 형식 사용 이유

Authorization 헤더에 API 키를 보낼 때, Bearer 토큰 형식을 사용하는 것은 관례다. 

Bearer 토큰은 서버가 '이 키를 가진 사용자는 요청을 보낼 권한이 있다'고 간주하게 만든다. 

따라서 API 키도 Bearer 토큰 형식으로 전송하는 것이 일반적이며, 많은 서버 구현에서 이를 지원한다.

이처럼 API 키는 간단하고 효과적인 인증 방식이지만, 높은 보안이 필요하다면 OAuth나 JWT 같은 더 복잡한 인증 방식을 고려해야 한다.
