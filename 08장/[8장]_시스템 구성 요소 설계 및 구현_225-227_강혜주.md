# 8.4.2 인가

사용자가 인증을 마쳤다면 이제 요청한 리소스에 접근할 수 있는 적절한 권한이 있는지 확인해야 한다.  
이를 **인가**라고 한다.

대규모 애플리케이션에서 어떤 방식으로 인가를 구현하는지 살펴보자.

- **역할 기반 접근 제어(Role-Based Access Control, RBAC)**
    - 역할 기반 접근 제어는 인가를 구현할 때 많이 사용하는 방법이다.
    - 권한을 사용자 개개인이 아니라 역할(role)에 할당한다.
    - 사용자는 특정 역할에 연결되며, 이 역할이 가진 권한에 따라 리소스 접근이 허용된다.
    - 개별 사용자 권한이 아니라 역할만 관리하면 되므로 관리가 훨씬 간단하다.

- **속성 기반 접근 제어(Attribute-Based Access Control, ABAC)**
    - 속성 기반 접근 제어는 인가를 속성(attribute)을 기반으로 정의하는 접근 제어 방식이다.
    - 이 속성은 사용자, 리소스, 환경 등 다양한 속성을 바탕으로 권한을 정의하며, 이런 속성들을 조합하기도 한다.
    - 예를 들어 사용자 부서(department), 리소스 유형(type), 요청된 시간(time) 등을 속성으로 사용할 수 있다.
    - 속성 기반 접근 제어는 역할 기반 접근 제어보다 더 세밀한 제어가 가능하지만, 구현이 상대적으로 더 복잡하다.

역할 기반과 속성 기반으로 접근 제어를 구현하는 방식은 대규모 시스템에서 주로 사용하는 방법이다.  
API 보안을 제대로 이해하려면 API 운영에서 보안 통신과 요청 빈도 제한이 왜 중요한지 이해해야 한다.

# 8.4.3 API의 보안 통신

API를 설계할 때 데이터 전송을 안전하게 보호하는 것이 중요하다.

이를 위해 기본적으로 **HTTPS(HyperText Transfer Protocol Secure)** 를 사용하는 것이 필수적이다.

HTTPS는 클라이언트와 서버 간 통신을 암호화하여 공격자나 도청자가 데이터를 가로채거나 해독하기 어렵게 만든다.  
특히 비밀번호, 개인 정보, 금융 거래 등 민감한 데이터를 다룰 때 매우 중요하다.

또 **TLS(전송 계층 보안)** 설정을 최신 상태로 유지하는 것도 중요하다.

보안 취약점이 새로 발견되거나 보안 표준이 발전함에 따라 TLS 설정을 최신화하면 새로운 보안 문제를 예방하고 시스템을 보호할 수 있다.

이는 암호화 및 데이터 무결성을 위한 업계 표준을 준수하는 데도 필요하다.

# 8.4.4 요청 속도 제한

요청 속도 제한은 API 관리와 보안에서 중요한 메커니즘이다.
특정 시간 내 클라이언트가 보낼 수 있는 요청 수를 제한함으로써 API가 과도하게 사용되는 것을 방지할 수 있으며, 특히 **서비스 거부(DoS) 공격** 방어에 효과적이다.

> DoS: 대량의 요청을 보내 서버를 과부하 상태로 만들어 서비스 품질 저하나 서비스 장애를 유발하려는 시도

요청 속도 제한을 적용하면 서버 자원을 효율적으로 관리하고, 사용자 간 자원 사용을 균형 있게 분배할 수 있다.
뿐만 아니라 특정 사용자나 애플리케이션이 API를 독점적으로 사용하는 것을 방지하고, 모든 사용자가 안정적으로 서비스를 이용할 수 있도록 할 수 있다.

## 요청 속도 제한 알고리즘

클라이언트의 요청 속도를 제한하는 알고리즘은 여러 가지가 있다.  
각각 장단점이 있으며, 대표적인 방식은 다음과 같다.

- **토큰 버킷(token bucket) 알고리즘**
    - 요청 수를 제한하는 방식을 버킷과 토큰으로 비유한 알고리즘이다.
    - 요청 하나를 처리할 때마다 토큰 하나가 사용된다.
    - 토큰은 일정한 간격으로 버킷에 채워진다.
    - 버킷에 토큰이 남아 있지 않다면 추가 요청은 거부된다.

- **누수 버킷(leaky bucket) 알고리즘**
    - 토큰 버킷 알고리즘과 유사하지만, 버킷 아래에 구멍이 있다고 가정한다.
    - 토큰은 일정한 속도로 추가되지만 동시에 일정한 속도로 새어 나간다.
    - 버킷이 가득 차면 새로운 요청은 거부된다.

>책에서는 누수 버킷을 설명하면서 “토큰은 일정한 속도로 추가되지만 동시에 일정한 속도로 새어 나간다”라고 표현하고 있다.
>다만 이 표현은 토큰 버킷과 누수 버킷 개념이 섞인 설명으로 보인다.
>토큰 버킷에서는 토큰이 일정한 속도로 추가되고, 요청이 들어올 때 토큰을 소비한다.  
반면 누수 버킷에서는 요청이 불규칙하게 들어오고, 버킷에 쌓인 요청이 일정한 속도로 처리되어 빠져나간다.
따라서 누수 버킷의 핵심은 “토큰이 일정한 속도로 추가되는 것”이 아니라, “불규칙하게 들어온 요청을 일정한 속도로 흘려보내는 것”이라고 이해하는 편이 더 자연스럽다. 정리하면 다음과 같다.

| 구분           | 토큰 버킷                | 누수 버킷              |
| ------------ | -------------------- | ------------------ |
| 버킷에 쌓이는 것    | 토큰                   | 요청 또는 요청량          |
| 일정하게 추가되는 것  | 토큰                   | 없음                 |
| 불규칙하게 들어오는 것 | 요청                   | 요청                 |
| 일정하게 나가는 것   | 토큰을 소비한 요청           | 버킷에 쌓인 요청          |
| burst 처리     | 토큰이 쌓여 있으면 허용        | 버퍼에 쌓은 뒤 일정 속도로 처리 |
| 목적           | 평균 요청량 제한 + burst 허용 | 처리 속도 평탄화          |

---

- **슬라이딩 윈도우(sliding window) 알고리즘**
    - 특정 시간 내에서 처리된 요청 수를 추적하는 방식이다.
    - 특정 시간 내에 요청 수가 설정한 한도를 초과하면 이후 요청은 거부된다.

어떤 알고리즘을 선택할지는 시스템 요구 사항에 따라 달라진다.

예를 들어 요청을 얼마나 구체적으로 제한할지, 요청이 과도하게 몰렸을 때 이를 얼마나 허용할지, 구현이 얼마나 복잡해질 수 있는지를 고려해야 한다.

## 옮긴이 노트: 예시로 알아보는 각 알고리즘

### 1. 토큰 버킷 알고리즘

토큰 버킷 알고리즘은 놀이공원의 인기 있는 놀이기구 앞에 줄을 섰을 때를 떠올리면 된다.

매시간 놀이기구를 탈 수 있는 **토큰**이 제한된 숫자만큼 배포된다.  
사람이 많으면 토큰이 빨리 소진되고, 새로운 토큰은 일정한 속도로 채워진다.

토큰이 다 떨어지면 줄을 설 수 없다.

현실 예시로 보면, 어린이 놀이방에 입장하려면 토큰을 받아서 입장해야 하는 상황과 비슷하다.  
시간이 지나며 새로운 토큰이 채워지지만, 잠시 기다린 후 다시 입장할 수 있다.

---

### 2. 누수 버킷 알고리즘

이번에는 수도꼭지를 떠올려 보자.

수도꼭지로 물, 즉 요청이 들어오는 속도는 일정하지만, 아래로 새는 물, 즉 허용된 요청의 속도도 일정하다.

수도꼭지에 물을 너무 많이 부으면 어떻게 될까?  
바닥으로 넘쳐 버린다.

현실 예시로 보면, 커피숍에 가서 주문할 때 주문을 받는 직원은 일정한 속도로만 처리할 수 있다.  
손님이 너무 많으면 줄이 길어지거나 결국 새로 온 손님은 줄을 서지 못하게 된다.

> 누수 버킷은 요청을 임시로 담아두는 버퍼 또는 대기 큐에 까워보인다.  
요청은 사용자 트래픽에 따라 불규칙하게 들어오지만, 버킷에서 빠져나가는 처리 속도는 일정하다.  
따라서 순간적으로 요청이 몰리더라도 서버에는 일정한 속도로 요청이 전달된다.  
다만 버킷의 크기는 제한되어 있으므로, 대기 가능한 요청 수를 초과하면 새로운 요청은 거부된다.

---

### 3. 슬라이딩 윈도우 알고리즘

슬라이딩 윈도우 알고리즘은 시간 단위로 얼마나 자주 요청이 들어오는지 체크한다고 생각하면 된다.
한 시간 동안 몇 번만 요청할 수 있다고 설정하면, 그 안에서 요청이 초과되었을 때 추가 요청은 거절된다.
`슬라이딩`이라는 이름처럼 시간 창이 조금씩 움직이며 요청을 계산한다.
현실 예시로 보면, 음료 자판기는 한 시간에 음료 10잔만 판매할 수 있다.  
11번째 손님은 “다음 시간을 기다려 주세요.”라는 메시지를 받게 된다.
