### API

API(Application Programming Interface)는 서로 다른 소프트웨어 구성 요소가 **서로 소통할 수 있도록 돕는 다리** 역할을 한다. 마이크로서비스와 분산 시스템이 중요해진 지금, 시스템이 여러 서비스로 잘게 쪼개지면서 그 사이를 잇는 API의 역할도 그만큼 커졌다. 이 장에서는 REST와 gRPC라는 두 가지 대표적인 API 설계 방식을 살펴보고, 각각의 장점과 활용 사례를 알아본다.

### API 보안

API는 외부에 노출되는 입구라서 **공격 대상이 되는 경우가 많다.** 그래서 철저한 보안 대책을 마련하고 적용하는 게 중요하다. 이 장에서 다루는 API 보안의 기본 원칙은 네 가지다.

- **인증(Authentication)**: 너 누구냐 — 요청자가 본인이 맞는지 확인
- **권한 관리(Authorization)**: 너 이거 해도 되냐 — 인증된 사용자가 그 리소스에 접근할 권한이 있는지 확인
- **안전한 통신**: HTTPS/TLS로 전송 구간 암호화
- **속도 제한(Rate Limiting)**: 한 클라이언트가 과도하게 때려서 시스템을 망가뜨리지 못하게 요청 횟수 제한

### 로깅 & 메트릭

로깅과 메트릭은 분산 시스템에서 **모니터링을 통한 문제 해결**에 꼭 필요한 도구다. 둘 다 "시스템을 들여다보는" 수단인데 보는 각도가 다르다.

- **로깅**: 시스템이 **어떻게 동작하고 있는지**를 기록. 이벤트 단위의 서술 정보.
- **메트릭**: **성능 상태**를 수치로 파악. CPU, 응답시간, 처리량 같은 정량 지표.

이 둘이 있어야 문제를 빠르게 찾아낼 수 있다. 이 장에서는 분산 로깅을 설계·구현하는 방법과 메트릭의 중요성을 알아본다.

### 알림(Alerting)

알림은 시스템 문제를 **사전에 파악하고 관리**하는 데 효과적이다. 문제가 커지기 전에 엔지니어에게 경고를 보내서 미리 조치를 취할 수 있게 해준다. 메트릭이 "지금 상태가 이렇다"를 보여주는 거라면, 알림은 "이 상태가 위험하니 너가 봐야 한다"고 깨워주는 역할. 이 장에서는 알림의 필요성, 효과적으로 알림을 설정하는 방법, 알림에 대응하는 방법을 알아본다.

### 트레이싱(Tracing)

마지막은 트레이싱이다. 트레이싱은 분산 시스템에서 **하나의 요청이 여러 서비스를 거치는 경로를 추적**하는 방법이다. 성능 문제(어디서 느려졌나)를 파악하거나 시스템이 어떻게 동작하는지 이해하는 데 유용하다.

> 로깅은 "한 서비스 안에서 무슨 일이 있었나", 트레이싱은 "여러 서비스를 가로지르는 요청 하나가 어디를 어떻게 거쳤나"다. 우리처럼 API → SNS/SQS → ES/Redis로 흐르고 멀티 리전까지 타는 구조에선, 로그만 봐선 요청 전체 흐름이 안 보여서 트레이싱이 특히 중요해진다.

## 8.1 REST API

REST(REpresentational State Transfer)는 네트워크 애플리케이션을 설계하는 아키텍처 스타일이다. 프로토콜이 아니라 스타일. 표준 HTTP 메서드를 그대로 쓰니까 구현이 단순하고 직관적이라 다들 쓴다.

### 8.1.1 설계 원칙

- **리소스 기반**: 객체/데이터/서비스 등 클라이언트가 접근하는 모든 걸 리소스로 봄.
- **URI로 식별**: 리소스는 URI로 구분.
- **HTTP 메서드 = CRUD**: `GET / POST / PUT / DELETE` ↔ `Read / Create / Update / Delete`. DB의 CRUD랑 역할이 거의 1:1.
- **무상태성(Statelessness)** ← 이게 핵심: 클라이언트가 요청 보낼 때 처리에 필요한 모든 정보를 **요청 안에 다 담아야 함**. 서버가 요청 간 상태를 기억할 필요가 없음.

> 무상태성이 왜 중요하냐?
>
> 서버가 상태를 안 들고 있으니 동작이 예측 가능하고, **수평 확장(스케일 아웃)**이 쉽다. 어느 서버로 요청이 가든 결과가 같음. 멀티 리전(KR/EU/AU)처럼 인스턴스가 여기저기 떠 있는 환경에서 특히 이득. 세션을 서버 메모리에 안 박고 토큰(JWT) 같은 걸로 요청마다 인증 정보를 실어 보내는 이유다.

### 8.1.2 활용 사례

- **공개 API에 적합**: HTTP만 쏠 수 있으면 됨 → 웹 브라우저, 모바일 앱, 다른 서버 등 클라이언트 가리지 않음.
- **CRUD 작업에 적합**: 데이터 생성/조회/수정/삭제가 주력인 시스템에 잘 맞음. 사용자·제품 목록 관리하는 웹 앱이 전형적인 케이스.

> 사실상 우리가 매일 짜는 Spring Boot + REST Controller가 딱 이거다. 모바일 앱(Flutter)이 클라이언트, 백엔드가 리소스 제공. CRUD 중심이라 REST가 정답에 가까움.

### 8.1.3 장단점

**장점**
- 구조가 간단하고 이해하기 쉬움.
- HTTP 메서드 + 상태 코드를 그대로 쓰니 HTTP에 익숙하면 매우 직관적.

**단점**
- 여러 요청을 한 번에 묶어서 처리해야 하는 복잡한 작업에선 비효율적 (요청이 잘게 쪼개짐).
- JSON으로 데이터 교환 → 데이터양 많아지면 **페이로드가 커짐**.

> 이 단점(요청 분할 비효율 + JSON 페이로드 비대)을 해결하려고 나온 게 다음 절의 **gRPC**다. 둘 다 이해하면 상황별로 골라 쓸 수 있음.
