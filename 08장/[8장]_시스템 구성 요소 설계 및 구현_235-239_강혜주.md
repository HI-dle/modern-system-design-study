# 8.8 분산 시스템에서 트레이싱

# 트레이싱
- 요청이 여러 서비스를 거쳐 처리되는 과정을 추적하는 기법
    - 요청이 처리되는 상세한 과정을 확인할 수 있고, 성능 문제를 진단하고 시스템 동작을 이해하는 데 유용한 도구로 쓰인다.
- 분산 시스템에서는 하나의 요청이 여러 서비스를 거치는 경우가 많으며, 특히 문제가 생겼을 때 이 요청이 어떤 경로로 처리되었는지 파악하기 어려운데, 이런 상황에서 트레이싱을 활용할 수 있다.
- 트레이싱으로 요청이 시스템에서 어떻게 처리되는지 상세히 볼 수 있다.
    - 서비스 간 상호 작용, 각 서비스의 지연 시간, 처리 중 발생한 오류 등을 파악할 수 있다.

> 트레이싱을 활용하면 성능 문제를 분석하고, 병목 지점을 파악하며, 시스템에서 요청이 처리되는 전체 흐름을 이해하는 데 유용하다.
> 트레이싱의 중요성, 트레이싱을 구현할 수 있는 오픈 소스 라이브러리, 분산 트레이싱을 설계할 때 따라야 할 모범 사례를 살펴본다.

---

# 8.8.1 분산 트레이싱

분산 트레이싱은 트레이싱 개념을 분산 시스템에 적용한 방식이다.

하나의 요청이 여러 서비스와 서버를 거치는 과정을 추적하는 기법이다.  
요청이 처리되는 각 단계를 스팬(span)으로 기록하며, 스팬이 모여 요청 전체의 흐름을 보여주는 트레이스(trace)를 구성한다.

## 장점

- **성능 최적화**
    - 요청 흐름을 시각화하면 성능 병목 현상을 파악하고 이를 최적화할 수 있다.

- **오류 진단**
    - 요청이 실패했을 경우 트레이스를 활용하여 오류가 발생한 위치와 원인을 파악할 수 있다.

- **시스템 동작 파악**
    - 트레이싱을 활용하면 시스템에서 요청이 처리되는 흐름을 명확히 파악할 수 있다.
    - 덕분에 새로운 팀원이 시스템 구조를 빠르게 이해할 수 있고, 시스템 설정이나 아키텍처 변경 계획이 있을 때도 유용하게 사용할 수 있다.

---

# 8.8.2 분산 트레이싱을 위한 오픈 소스 라이브러리

분산 트레이싱을 구현할 때 사용할 수 있는 오픈 소스 라이브러리도 있으며, 저마다 고유한 특징과 기능이 있다.

## 예거(Jaeger)

- 예거는 우버에서 개발한 분산 트레이싱 시스템으로, 오픈 소스로 공개된 이후 마이크로서비스 기반 분산 시스템의 모니터링과 문제 해결에 널리 활용된다.
- 구글의 Dapper와 OpenZipkin에서 영감을 받아 설계되었다.
- 분산 컨텍스트 전파, 트랜잭션 모니터링, 문제의 근본 원인 분석, 성능 최적화 같은 기능을 갖추고 있다.
- 엘라스틱서치, 카산드라, 카프카 같은 다양한 스토리지 백엔드를 활용하여 확장성을 확보할 수 있다.

## 집킨(Zipkin)

- 집킨은 마이크로서비스 아키텍처에서 발생하는 지연 문제를 분석하고자 각 서비스의 요청 처리 시간을 수집하고 관리하는 분산 트레이싱 시스템이다.
- 간단하고 직관적인 인터페이스로 요청 흐름을 시각화할 수 있어 개발자가 성능 병목 현상을 빠르게 파악하고 문제를 해결할 수 있다.
- 집킨은 메모리, MySQL, 카산드라, 엘라스틱서치 등 다양한 데이터 저장소를 지원하며, 여러 블로그의 언어 및 프레임워크와 쉽게 연동할 수 있다.

## 오픈텔레메트리(OpenTelemetry)

- 오픈텔레메트리는 애플리케이션에서 분산 트레이스와 메트릭 데이터를 수집할 수 있는 API, 라이브러리, 에이전트, 수집기 서비스를 통합적으로 제공하는 관측 프레임워크이다.
- 여러 플랫폼과 도구에서 텔레메트리 데이터를 표준화하여 수집하고 분석하도록 설계되었다.
- 다양한 프로그래밍 언어를 지원하고, 프로메테우스와 예거 같은 관측 도구와도 쉽게 연동할 수 있다.
- 또 컨텍스트 전파, 분산 트레이싱, 메트릭 수집 같은 고급 기능도 있어 클라우드 네이티브 애플리케이션에서 폭넓게 활용할 수 있다.
## 트레이싱 대시보드에서 보는 요청 흐름

![[8-4.png]]

이 그림은 하나의 API 요청이 분산 시스템 내부에서 어떻게 처리되는지 시간 순서대로 보여준다.

사용자가 브라우저에서 요청을 보내면, 요청은 백엔드에 진입한 뒤 여러 서비스와 프로세스를 거쳐 처리된다.  
트레이싱 대시보드는 이 전체 과정을 하나의 트레이스로 묶고, 각 처리 단계를 스팬(span)으로 나누어 보여준다.

- **트레이스(trace)**
    - 하나의 요청이 시작되어 끝날 때까지의 전체 흐름
    - 예를 들어 사용자가 브라우저에서 API를 호출하고 응답을 받기까지의 전체 과정

- **스팬(span)**
    - 트레이스 안에서 각각의 작업 단위
    - 예를 들어 `Service A 진입`, `DB 조회`, `Service B 호출`, `외부 API 호출` 같은 개별 처리 구간

- **트레이스 시간**
    - 요청 전체가 처리되는 데 걸린 시간
    - 가장 바깥쪽 긴 막대가 전체 요청 시간을 의미한다.

- **루트 스팬 시간**
    - 요청이 시스템에 처음 들어온 뒤, 최상위 작업이 완료될 때까지의 시간
    - 보통 브라우저 요청이나 최초 백엔드 진입 구간이 루트 스팬이 된다.

- **백엔드 처리 시간**
    - 실제 백엔드 서비스 내부에서 요청을 처리하는 데 걸린 시간
    - 네트워크 대기, 서비스 간 호출, DB 접근, 외부 API 호출 등이 포함될 수 있다.

- **서비스 진입**
    - 요청이 특정 서비스에 들어온 시점
    - 예를 들어 `Service A`가 요청을 받고 처리하기 시작한 구간이다.

- **프로세스 종료 스팬**
    - 특정 프로세스 또는 작업이 끝나는 구간
    - 예를 들어 DB 조회, 외부 데이터 저장소 접근, 내부 처리 완료 등이 여기에 해당한다.

# 8.8.3 분산 트레이싱 구현 모범 사례

분산 시스템에서 트레이싱을 구현하는 일도 만만치 않은 작업이다.  
하지만 다음 모범 사례를 참고하면 보다 효과적으로 구현할 수 있다.

- **코드에 트레이싱 추가하기**
    - 트레이스를 수집하려면 코드에 트레이싱 라이브러리를 적용해야 한다.
    - 보통 추적하고자 하는 작업에 스팬을 설정하는 작업이 필요하다.

- **컨텍스트 전파하기**
    - 트레이스를 여러 서비스에 걸쳐 추적하려면 트레이스 컨텍스트를 서비스 간에 전달해야 한다.
    - 보통 HTTP 요청 헤더에 컨텍스트를 담아 전달하는 방식으로 처리한다.

- **스팬 이름을 명확하게 작성하기**
    - 스팬 이름은 어떤 작업을 나타내는지 쉽게 파악할 수 있도록 구체적이고 일관성 있게 정해야 한다.

- **스팬에 주석 추가하기**
    - 스팬에 추가적인 메타데이터를 포함하면 해당 작업 맥락을 더 잘 이해할 수 있고 문제를 분석하고 해결하기 쉽다.

# 8.8.4 분산 트레이싱 모범 사례

이 장에서는 gRPC API, API 보안, 메트릭, 로깅, 알림, 분산시스템에서 트레이싱 등 시스템 설계에서 여러 가지 중요한 요소를 다뤘다  
이런 요소는 모두 안정적이고 효율적이며 확장 가능한 시스템을 만드는 데 핵심적인 역할을 한다.

각 요소를 어떤 방식으로 활용할 수 있는지 몇 가지 모범 사례로 알아보자.

- **적절한 API 설계 선택하기**
    - REST와 gRPC API에는 각각 장점과 적합한 활용 용도가 있다.
    - 성능, 사용 편의성, 호환성 등 시스템 요구 사항에 가장 적합한 방식을 선택하는 것이 중요하다.

- **API 보안을 최우선으로 고려하기**
    - 강력한 인증 및 권한 부여 메커니즘을 구축하고, HTTPS로 데이터 전송 보안을 강화해야 한다.
    - 또 요청 속도 제한을 설정하여 과도한 요청이나 악의적인 접근 시도를 효과적으로 차단해야 한다.
    - 이런 조치는 API의 안정성과 신뢰성을 유지하는 데 필수적이다.

- **중앙 집중형 로깅 구현하기**
    - 모든 서비스에서 생성된 로그를 중앙 위치에 수집하고 저장해야 한다.
    - 이렇게 하면 로그를 더 쉽게 검색하고 분석할 수 있으며, 시스템 동작에 대한 유용한 정보를 얻을 수 있다.

- **핵심 메트릭 모니터링**
    - 시스템, 애플리케이션, 비즈니스와 관련된 주요 메트릭을 파악하고 이를 주기적으로 모니터링하면, 시스템 성능을 정확히 파악하고 데이터 기반의 의사 결정을 내릴 수 있다.

- **구체적이고 실행성 있는 알림 설계하기**
    - 알림은 실제 문제를 신속히 포착할 수 있도록 적절한 민감도를 유지하면서도 알림 피로를 최소화하도록 설계해야 한다.
    - 문제를 파악하는 데 필요한 정보를 충분히 담고, 알림이 의도한 대로 작동하는지 정기적으로 점검해서 신뢰성을 확보하는 것도 중요하다.

- **분산 트레이싱 활용하기**
    - 분산 트레이싱을 구현하여 요청이 어떤 방식으로 서비스를 거치는지 살펴봄으로써 성능 문제를 진단하고 요청 흐름을 명확히 파악하여 시스템 성능을 최적화하는 데 도움을 받을 수 있다.

시스템 설계 목표는 단순히 작동하는 시스템을 만드는 것이 아니라, 다양한 상황에서 안정적으로 작동하고 필요에 따라 확장할 수 있는 시스템을 만드는 것이다.

# 8.9 요약

이 장에서는 분산 시스템을 설계하고 운영할 때 중요한 기본 요소들을 살펴보았다.

주요 내용은 다음과 같다.

- **API 설계**
    - REST와 gRPC의 특징과 적합한 사용 사례를 비교했다.
    - 시스템 요구사항에 따라 적절한 API 방식을 선택하는 것이 중요하다.

- **API 보안**
    - 인증, 인가, HTTPS/TLS 기반 보안 통신, 요청 속도 제한을 다루었다.
    - 안전한 API 운영을 위해 접근 제어와 트래픽 제어가 필요하다.

- **로깅과 메트릭**
    - 분산 시스템을 모니터링하고 디버깅하기 위해 중앙 집중형 로깅과 핵심 메트릭 수집이 중요하다.
    - 이를 통해 시스템 상태를 파악하고 문제 원인을 분석할 수 있다.

- **알림**
    - 시스템 문제를 빠르게 감지하고 대응하기 위한 알림 설계가 필요하다.
    - 알림은 너무 민감하지 않으면서도 실제 문제를 놓치지 않도록 설계해야 한다.

- **트레이싱**
    - 하나의 요청이 여러 서비스를 거치는 흐름을 추적하는 방법을 다루었다.
    - 트레이싱을 통해 병목 구간, 오류 발생 위치, 서비스 간 호출 흐름을 파악할 수 있다.

이 장은 안정적이고 확장 가능한 분산 시스템을 만들기 위해 필요한 API 설계, 보안, 관측성 요소들을 다뤘다.  
이러한 요소들은 시스템을 운영하면서 문제를 빠르게 발견하고, 성능을 개선하며, 신뢰성 있는 서비스를 제공하는 데 중요한 기반이 된다.
