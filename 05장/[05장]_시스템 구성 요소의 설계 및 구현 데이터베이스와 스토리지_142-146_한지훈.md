## 5.4 get 및 put 함수 구현

- 이 절에선 키-값 저장소에서 get, put 함수를 구현하는 방법을 살펴보자.

### 5.4.1 get 및 put 함수 구현

 - 시스템간 설계의 핵심 요구 사항 중 하나는 구성 가능성이다.
 - 가용성, 일관성, 비용 효율성과 효율성 간의 군형을 조정할 수 있는 능력을 의미한다. 이런 구성 가능성을 확보 하는데 키-값 저장소의 기본 기능인 get, put 연산을 통합할 수 있다.
 - 시스템에서 모든 노드가 get(읽기), put(쓰기) 연산을 수행할 수 있다. 이런 연산을 담당하는 노드를 코디네이터라 한다. 이는 보통 우선 목록에서 상위 n개의 노드 중 첫 번째에 해당한다.
 - 클라이언트는 2가지 방식으로 노드를 선택할 수 있다.
   - 요청을 일반 로드 밸런서로 라우팅하는 방식
   - 분할 인식 클라이언트 라이브러리를 사용해 요청을 해당 코디네이터 노드로 직접 전달하는 방식

 - 두 방식 모두 장점이 있다. 첫 번째 방식은 클라이언트가 코드에 종속되지 않도록 하며 두 번째 방식은 클라이언트가 특정 서버에 직접 접근할 수 있어 홉 수가 줄어들어 지연 시간을 낮출 수 있다.
 - 서비스 유연성을 높이려면 가용성, 일관성, 비용 효울성, 성능 간 균형을 조정할 수 있어야 한다. 쿼럼 시스템에서 사용하는 프로토콜과 유사한 방식을 활용하면 이를 실현할 수 있다.
 - 우선 목록의 상위 n값이 3이라면 이는 데이터 복사본 3개로 유지해야 함을 의미한다. 노드가 링 구조로 배치되고 노드 A,B,C,D,E가 시계 방향 순서로 나열되어 있다고 할 때 쓰기 작업이 노드 A에서 수행되면 데이터 복사본은 시계 방향으로 이동해 노드 B와 노드 C에 저장된다. 이는 링 구조에서 시계 방향으로 이동하며 다음 노드에 저장하는 방식이다.

### 5.4.2 r과 w 사용

 - r과 w란 두 변수를 생각 해볼떄 r은 읽기 작업이 성공하는 데 필요한 최소 노드 개수를 의미 w는 쓰기 작업이 성공하는 데 필요한 최소 노드 개수를 나타낸다. 예시로 r = 2고 데이터가 노드 3개에 저장되어 있다면 시스템은 노드 두 개에서 데이터를 읽어 온다. r과 w의 값을 설정할 때는 읽기와 쓰기 작업이 최소 하나의 노드에서 공통으로 교촤디ㅗ도록 해야한다. 이것으로 읽기 작업이 항상 최신 쓰기 데이터를 접근할 수 있게 보장할 수 있다. 이를 위해서는 r + w > n 이라는 조건을 반족해야 한다.


| n | r | W | 설명 |
|---|---|---|------|
| 3 | 2 | 1 | 제약 조건 위반: r + w > n 충족하지 않음 |
| 3 | 2 | 2 | 제약 조건 충족 |
| 3 | 3 | 1 | 느린 읽기와 빠른 쓰기 |
| 3 | 1 | 3 | 느린 쓰기와 빠른 읽기 |


 - 클라이언트에서 들어오는 읽기 요청이 성공하려면 더 많은 노드가 요청에 응답하도록 설계해야 한다. 이런 설계 때문에 읽기와 쓰기 요청에서 속도가 느려진다. 이는 쓰기 요청에서도 마찬가지고 읽기와 쓰기 성능을 저해하지 않으며 임의의 노드 장애를 견딜 수 있을 만큼 충분한 중복성을 확보하려면 읽기와 쓰기의 균형을 맞춘  쿼럼 설정이 필요하다.
 - n이 3인 경우 데이터가 복제되는 노드가 3개고 w가 2인 경우 두 노드에 데이터를 쓰는 것으로 요청이 성공적으로 처리된다. 3번쨰 노드는 데이터를 비동기적으로 업데이트한다.
 - 이 모델에서는 get 연산의 지연 시간이 레플리카 r개 중 가장 느린 레플리카로 좌우된다. 이는 r 값이 클수록 일관성보단 가용성을 우선시하기 때문이다.

## 5.5 키-값 저장소의 장애 허용성과 장애 식별

- 분산 시스템에서 장애를 처리하는 일반적인 방법 중 하나는 **쿼럼 기반 시스템**을 사용하는 것이다. 쿼럼은 분산 트랜잭션이 작업을 수행하는 데 필요한 최소 투표수를 의미한다.
- 그러나 합의 과정에 참여하는 서버 중 하나가 장애를 일으키면 작업이 진행될 수 없으며, 이는 시스템의 가용성과 내구성에 영향을 미칠 수 있다.

- 엄격한 쿼럼 멤버십 대신 느슨한 쿼럼을 사용하는 방식을 생각할 수도 있다.
- 일반적으로 리더가 합의에 참여하는 구성원 간 통신을 조율하며, 쓰기 작업이 성공적으로 완료되면 참여자들이 리더에게 확인 메시지를 전송한다. 리더는 이런 확인 메시지를 받은 후 클라이언트에 응답한다.
- 하지만 이 구조는 네트워크 장애에 취약하다. 리더가 일시적으로 다운되거나 참여자들이 리더와 통신할 수 없으면 더 이상 유효하지 않다고 판단하여 리더가 '죽었다'고 선언한다.
- 이 때문에 새로운 리더를 선출해야 하며, 빈번하게 리더를 선출하면 시스템이 작업 처리보다 리더를 선택하는 데 더 많은 시간을 소모하게 되어 성능 저하로 이어질 수 있다.

- 느슨한 쿼럼에서는 우선 목록에 있는 노드 중 정상적으로 동작하는 노드 n개가 모든 읽기 및 쓰기 작업을 처리한다. 다만 이 정상 노드 n개는 일관된 해시 링에서 시계 방향으로 이동하며, 처음 식별된 노드 n개와 정확히 일치하지 않을 수 있다.

- n이 3인 상황에서 쓰기 작업 중 노드 A가 일시적으로 사용 불가능하거나 접근할 수 없을 때 시스템은 우선순위 목록에서 정상적으로 작동 중인 노드를 찾아 요청을 전송한다. 이때 요청은 노드 D로 전달되어 요구되는 가용성과 내구성을 보장할 수 있다.
- 요청 처리가 완료된 후 노드 D는 원래 요청 대상이었던 노드 A를 위한 힌트를 포함한다. 여기서 힌트를 포함한다는 것은 노드 D는 데이터를 임시로 저장했을 뿐이며 이 데이터는 원래 노드 A에 저장해야 한다는 정보를 함께 기록한다는 의미이다.
- 이후 노드 A가 다시 정상 상태로 복구되면 노드 D는 해당 요청 정보를 노드 A로 전송하여 데이터를 업데이트한다. 전송이 완료되면 노드 D는 로컬 저장소에서 관련 데이터를 삭제하여 시스템 내 레플리카의 총수를 유지한다.

- 이 상황에서 데이터가 흐르는 과정을 나타낸다 처음엔 쓰기 요청이 노드 A로 전달되지만 노드 A가 일시적으로 접근할 수 없는 상태가 되면 요청이 노드 D로 넘어간다. 노드 D는 요청을 처리하는 동시에 이 요청이 원래 노드 A로 가야 했던 정보임을 함께 저장한다. 이후 노드 A가 복구되면 노드 D는 노드 A를 대신해 처리했던 모드 요청을 노드 A로 전달하고 임시로 저장했던 데이터를 삭제하여 내부 저장소를 초기 상태로 정리한다. 이것은 힌트 전달 방식이라고도 하며 일시적인 노드 장애가 발생하더라도 읽기와 쓰기 작업을 문제없이 처리하도록 한다.


### 5.5.2 영구적 장애 관리

- 노드에 영구적인 장애가 발생했을 때 시스템 내구성을 강화하려면 레플리카를 동기화 상태로 유지하는 것이 중요하다. 핵심 목표는 레플리카 간의 불일치를 빠르게 감지하고 데이터 전송을 최소화하는 것이다. 머클 트리(merkle tree)를 사용하면 이 문제를 해결할 수 있다.

 - 머클 트리는 각 키-값 쌍을 해싱하여 트리의 리프 노드(leaf node)로 사용한다. 트리의 상위 부모 노드는 자식 노드의 해시 값을 저장하여 데이터 간 관계를 나타내는 계층적 구조를 만든다.
 - 머클 트리의 특징은 각 가지(branch)를 개별적으로 확인할 수 있어 전체 트리나 데이터셋을 모두 비교하지 않고도 데이터 불일치를 효과적으로 검증할 수 있다는 점이다.
 - 이 방식은 레플리카 간 데이터 불일치를 확인할 때 전송해야 하는 데이터양을 크게 줄여 준다.
   - 두 트리에서 루트 노드의 해시와 리프 노드가 동일하다면 추가적인 동기화는 필요 없다.
   - 불일치한다면 호스트는 자식 노드의 해시 값을 비교하며 동기화가 필요한 키를 파악하고 이를 반복하여 트리의 리프 노드까지 도달한다.
 - 이런 안티 엔트로피(anti-entropy) 메커니즘은 동기화 과정에서 데이터 전송량과 저장소 접근을 최소화하면서도 레플리카 간 데이터 일관성을 효과적으로 보장한다.

- 머클 트리의 동작 방식을 한번 살펴보자.
   - 모든 키를 해싱하여 리프 노드를 생성한다.
   - 각 노드는 자신에게 할당된 키 범위에 대해 각 가상 노드마다 고유한 머클 트리를 유지한다. 즉, 분산 시스템에서 데이터는 여러 가상 노드에 나누어서 저장되는데, 하나의 물리적 노드가 이런 가상 노드 중 일부를 관리하며 해당 범위의 데이터를 검증하고 동기화하는 데 사용하는 것이 머클 트리이다. 이렇게 함으로써 특정 노드가 관리하는 데이터 범위 내에서 키의 정확성을 효율적으로 검증할 수 있다.
   - 두 노드는 공통된 키 범위에 해당하는 머클 트리의 루트 해시 값을 교환한다. 비교 과정은 다음과 같다.
     - a. 머클 트리의 루트 노드 해시를 비교한다.
     - b. 두 해시가 동일하다면 더 이상 진행하지 않는다.
     - c. 다르다면 재귀적으로 왼쪽과 오른쪽 자식 노드를 탐색하면서 차이점을 찾아 동기화한다.

 - 머클 트리의 장점은 전체 트리나 데이터셋을 불러오지 않고도 각 가지를 개별적으로 검증할 수 있다는 점이다. 동기화 과정에서 교환해야 할 데이터양은 줄어들고 안티 엔트로피 과정에서 필요한 저장소 접근 횟수도 감소한다.
 - 하지만 시스템에 새로운 노드를 추가하거나 기존 노드를 삭제하면 각 노드가 담당하는 키 범위가 바뀌면서 여러 키 범위에 영향을 주는데, 이에 따라 트리 해시를 다시 계산해야 하는 것은 단점이다.
 - 또 해시 링 내 다른 노드들이 장애를 감지할 수 있도록 설계해야 한다.
