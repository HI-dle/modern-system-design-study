# 관계형 데이터베이스와 비관계형 데이터베이스의 장단점

## 관계형 데이터베이스
- 장점
  - 데이터 중복 처리: 
    - 테이블 간 데이터 중복을 최소화하여 데이터 무결성을 유지
    - 저장 공간을 최적화하고 데이터 검색을 간편
  - 강력한 보안 기능: 
    - 내장된 보안 기능을 이용하여 허가되지 않은 접근을 차단으로 유출 방지
  -  ACID 트랜잭션: 
    - 기본적으로 ACID를 지원하여 작업 중 데이터의 일관성과 신뢰성을 유지
- 단점
  - 성능: 
    - 여러 테이블에서 복잡한 조인을 거쳐 데이터를 가져올 경우 성능 저하가 발생.
  - 메모리 소모량: 
    - 행과 열이 저장 공간을 차지하며, null 값이 있어도 메모리 사용량이 증가합니다.
  - 복잡성: 
    - 여러 테이블 간 관계와 조인을 처리하는 작업은 쉬운 일이 아니어서 데이터베이스 운영에 어려움을 줄 수 있다.
  - 수평적 확장의 어려움: 
    - 기존 관계형 데이터베이스는 수평적 확장이 어려워 대용량 데이터를 처리하기에 적합하지 않음


## 비관계형 데이터베이스

- 장점
  - 유연한 데이터 모델: 
    - 비관계형 데이터베이스는 정형 데이터와 비정형 데이터를 모두 효과적으로 저장 가능 
    - 다양한 데이터 형식을 다루고 데이터 구조 변화에도 유연하게 대응 가능.
  - 스키마 업데이트: 
    - 스키마를 수정해도 애플리케이션에 영향을 주지 않아 데이터 모델을 유연하게 변경 가능
  - 수평적 확장: 
    - 뛰어난 수평적 확장 기능으로 데이터와 작업량이 늘어나도 무리 없이 대응
- 단점
  - 부족한 표준화: 
    - 관계형 데이터베이스에 비해 설계 방식과 쿼리 언어가 제각각
    - 서로 다른 시스템 간 호환성이나 데이터 일관성을 유지하기 어려 울 수 있다.
    - ACID 트랜잭션 미지원: 
      - 대부분의 비관계형 데이터베이스는 특정 유형을 제외하고 ACID 트랜잭션을 제공하지 않아 일관성이 중요한 데이터 작업에서 신뢰성과 무결성을 유지하기 어렵

# 5.2 키-값 저장소

- 높은 확장성
- 간단하고 효율적인 데이터 저장 가능
- 배울내용
  - 설계하고 관리하는 데 필요한 개념
  - 키값 저장소의 요구 사항을 정의하고 API를 설계하는 과정
  - 확장성을 높이기 위해 일관된 해싱과 데이터 분할 복제 전략 같은 기법
  - 버전 관리를 하면서 데이터 변경으로 발생할 수 있는 충돌을 해결하는 방법 
  - 키-값 저장소의 장애 허용성을 높이고 오류 감지 메커니즘을 설계하는 방법

## 개념
- 키-값 저장소 또는 키-값 데이터베이스는 각 고유한 키가 특정 값과 연결되는 간단한 데이터 저장 방식
- 키를 사용하여 데이터를 저장·검색 업데이트 가능. 
- 기본 데이터 저장단위: 키-값 쌍 
- 단순한 구조 덕분에 읽기와 쓰기 작업의 효율성이 높다. 
- 키-값 저장소가 분산 시스템에서 유용한 이유
  - 분산
    - 키-값 저장소는 높은 성능과 확장성, 사용 편의성을 갖추고 있어 여러 분산 시스템에서 많이 사용.
    - 확장성: 
      - 키-값 쌍의 단순한 구조 덕분에 데이터를 여러 노드에 쉽게 분산할 수 있어 시스템의 처리량을 높일 수 있다.
    - 성능: 
      - 키-값 저장소는 데이터의 읽기와 쓰기 속도가 매우 빠르다. 
      - 특히 키-값 쌍이 같은 노드에 있을 때는 처리 속도가 더욱 빠르다.
    - 유연성: 
      - 관계형 데이터베이스와 달리 키-값 저장소는 고정된 데이터 스키마가 필요하지 않다. 
      - 정형, 반정형, 비정형 등 다양한 형태의 데이터를 저장할 수 있다.
    - 장애 허용성: 
      - 키-값 저장소는 데이터를 여러 노드에 복제하고 분할하도록 설계할 수 있다. 
      - 특정 노드에 장애가 발생하더라도 데이터에 접근할 수 있다.
      
- 정리: 
  - 분산 시스템에서 효율적이고 유연한 데이터 저장과 검색이 가능
  - 현대 소프트웨어 아키텍처의 핵심 요소.

> 노트
>  - 정형, 비정형, 반정형 데이터는 무엇인가요?
>    - 정형 데이터: 
>      - 학교의 성적표처럼 정해진 표 형식의 데이터. 
>        - 이름, 학번, 성적 등 특정한 항목이 있고, 모든 학생에 대해 동일한 형식으로 정리된 데이터
>    - 반정형 데이터: JSON 파일이나 XML 같은 형식이 이에 해당. 
>      - ex. 영화 사이트의 영화 정보 페이지. 
>        - 영화마다 제목, 감독, 배우 등이 존재, 
>        - 한 영화에는 줄거리 정보가 있고 다른 영화에는 없는 경우도 있다. 
>        - 일정한 형식이 있지만 완전히 고정되지 않은 데이터를 의미. 
>    - 비정형 데이터: 
>      - 자유롭게 작성한 글이나 사진처럼 정해진 형식이 없는 데이터. 
>      - 예를 들어 우리가 친구와 주고 받는 대화 메시지나 사진 파일, 동영상 같은 것들이 비정형 데이터.
>   
> 
> ### 정형 데이터 예시: 표 형태
> 
> ```python
> import pandas as pd
> 
> structured_data = pd.DataFrame({
>     '이름': ['Alice', 'Bob'],
>     '나이': [24, 27],
>     '직업': ['Engineer', 'Doctor']
> })
> 
> print(structured_data)
> ```
> 
> ### 반정형 데이터 예시: JSON 형태
> 
> ```python
> semi_structured_data = {
>     "사용자": [
>         {"이름": "Alice", "나이": 24, "직업": "Engineer"},
>         {"이름": "Bob", "나이": 27, "직업": "Doctor"}
>     ]
> }
> ```
> ### 비정형 데이터 예시
> 
> ```python
> unstructured_data = [
>     "Alice는 공학자입니다.",
>     "사진 경로: /images/bob_profile.jpg"
> ]
> ```
> 
> 이처럼 데이터 형태가 다양하므로, 유연한 저장 방식을 제공하는 키-값 저장소가 이를 효과적으로 다룰 수 있다.

- 데이터 기반 의사 결정의 중요성
- 데이터베이스와 스토리지 역할
  - 정보를 효율적으로 수집, 저장, 검색, 관리하는 능력
- 체계적으로 관리하고 안전하게 보존하는 데 필요한 기본 개념, 전략, 기술
- 데이터베이스와 스토리지 시스템의 설계 방식
- 데이터베이스
  - 데이터 구성: 데이터베이스는 데이터를 구조적이고 체계적으로 저장하는 방식 (테이블, 행, 열 형태로 쉽게 분류하고 접근)
  - 데이터 검색: 데이터베이스는 빠르고 효율적으로 데이터 검색이 가능
  - 데이터 무결성:
    - 제약 조건, 관계, 검증 규칙을 이용하여 데이터 무결성을 유지
    - 데이터의 신뢰성, 정확성, 일관성이 보장
  - 데이터 보안: 사용자 인증, 권한 부여, 암호화 등 보안 기능
  - 데이터 일관성: 트랜잭션 관리와 ACID(원자성, 일관성, 독립성, 지속성) 특성
  - 확장성: 
    - 대규모 데이터를 처리하고 데이터 수요가 증가함에 따라 확장
    - 단일 서버에 리소스를 추가하는 수직 확장과 여러 서버나 노드를 추가하는 수평 확장 모두 가능
  - 다중화, 백업 및 복구:
  - 복잡한 쿼리
  - 데이터 관계
  - 데이터 이력
  - 데이터 분석
  - 데이터 공유
- 데이터베이스 유형
  - 서로 다른 모델과 데이터 조회 메커니즘을 사용
  - 선택 기준: 요구 사항, 데이터 구조, 확장성, 성능 등 요소
  - 관계형
    - 테이블, 행, 열, 키, 정규화, SQL, ACID, 트랜잭션
  - NoSQL(비관계형)
    - 키값
    - 문서지향
      - json,xml
      - mongoDB, CouchDB, RavenDB
    - 컬럼 패밀리
      - 컬럼 패밀리 라는 그룹으로 데이터 구성
      - 수평확장 유리
    - 그래프기반
      - 데이터 간 관계 표현, 탐색하는 데 그래프 구조 사용
      - Neo4j, 아마존 Neptune, OrienntDB
    - 요구사항
      - 기능적: put, get, delete
      - 비기능적: 화장성, 성능, 내구성, 일관성, 가용성, 파티션 허용성
