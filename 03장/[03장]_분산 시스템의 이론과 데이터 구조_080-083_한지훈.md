# 분산 시스템의 이론과 데이터 구조

---

### 일관된 해싱

- 일관된 해싱은 분산 시스템에서 데이터를 여러 노드에 효율적으로 분배하며 노드를 추가하거나 삭제할때 데이터 위치를 다른 곳으로 옮겨야 하는 과정이 최소화되도록 하는 방식이다. 이 방식은 시스템의 확장성과 장애 허용성을 높이는 데 도둠이 된다.
- 해시 함수를 떠올리면 보통 모듈로 해싱 방식을 떠올린다. 이 방식은 데이터를 저장할 수 있는 노드나 버킷의 수가 고정되어 있다는 측징이 있다. 노드를 추가하거나 삭제하면 데이터 키를 노드에매핑하는 해시 함수가 변경되므로 많은 데이터를 다시 매핑하고 재분배 해야한다. 이 과정은 시간이 오래 걸리고 많은 리소스를 소모하며 시스템가용성을 떨어트릴 위험이 있다.

> 모듈로 해싱이란
> 
> 모듈로 해싱은 데이터 키를 해시 값으로 변환 후 노드 개수로 나눈 나머지를 사용해 데이터를 특정 노드에 할당하는 방식이다. 예시로 노드가 5개고 키 값이 17이라면 해시 값을 5로 나눈 나머지인 2를 사용해 두 번째 노드에 데이터를 저장한다.
> 

```javascript
// 키 값을  해시로 변환
function hashFunction(key) {
    return key.toString().split("").reduce((acc, char) => acc + char.charCodeAt(0), 0);
}

//모듈로 해싱: 해시 값을 노드 개수로 나눈 나머지로 늑정 노드 선택
function getNode(key, numNodes){
    const hash = hashFunction(key);
    return hash % numNodes;
}

//예시
const key = 17; //키 값
const numNodes = 5; //전체 노드 개수
console.log('Key ${key}는 노드 ${getNode(key, numNodes)}에 저장됩니다.')

```

- 일관된 해싱이 해결하고자 하는 주요 문제는 분산 시스템의 확장성과 장애 허용성이다. 이 기법은 시스템에 노드를 추가하거나 제거할 때 전체 데이터를 다시 매핑하지 않고 일부 데이터만 이동하도록 해 영향을 최소화 하는 것이 목표이다. 이 특성 덕분에 일괄된 해싱은 콘텐츠 전송 네트워크나 분산 데이터베이스처럼 노드의 추가와 제거가 빈번한 대구모 시스템에서 특히 유용하다.

- 일관된 해싱에서 해시 함수는 데이터 키와 노드 식별자를 공통 해시 공간에 매핑하는 데 사용된다. 이 해시 공간은 보통 원형 형태의 링 모양으로 표현되어 순환 구조를 이룬다. 각 노드를 식별자에 따라 링 위 특정 위치에 놓고 데이터를 저장하거나 가져올 때도 같은 해시 함수를 데이터 키에 적용해 링 위 특정 위치로 매핑한다. 그 위치에서 시계 방향으로 이동하면서 처음 만나는 노드가 해당 데이터를 저장하거나 처리하는 역할을 밭는다. 이 노드를 담당 노드 혹은 관리 노드라 한다.
- 예시로 클라이언트에서 API 요청이 들어오고 이를 서버에서 할당해야 하는 상황에서 시스템 내에는 sever0부터 server3까지 서버가 총 4개 있다고 가정하고 해시를 링으로 표현할때 서버 ID를 받아 해시 링에서 위치 값을 반환하는 해시 함수를 만들어야 한다. 이 함수로 다른 서버의 ID도 같은 방식으로 링 위의 특정 위치에 놓고 이 해시 함수를 h()라고 해보자.
- 서버 ID를 h() 함수에 전달해 링 위의 특정 좌표에 매핑한 값을 s라 가정한다면 다음과 같이 표현할 수 있다.
  - h(server0) = s0
  - h(server1) = s1
  - h(server2) = s2
  - h(server3) = s3
- API 요청도 해시 함수를 써서 매핑할 수 있다.
  - h(req_id1) = r1
  - h(req_id2) = r2 ...
- 어떤 서버가 어떤 요청을 처리할지 결정하기 위해 요청(req_id1)에ㅔ 해당하는 점(r1)을 가져오고 시계 방향으로 이동해 처음 만나는 서버 점을 찾는다. 이때 처음 만나는 서버 점은 s1로 이는 server1에 해당한다.

 - 위 그림과 같이 링 모양으로 해싱을 표현한 버전에는 몇가지 문제점이 있다.
 - 그림만 보면 각 서버 사이 API 요청이 고르게 분배 되는 것처럼 보이지만, 실제 상황에서는 요청이 균등하게 분배되지 않을 수 있다.
 - 서버 하나가 다운되면 시계 방향으로 다음 서버가 고장난 서버의 작업까지 해야한다. 이 때문에 해당 서버에 과부하가 걸릴 수 있고 결과적으로 서버가 다운될 수 있다. 이렇게 되면 과부하로 인한 연쇄 반응이 발생해 모든 서버가 차례대로 다운될 가능성이 있다.

- 위 그림에서 표현하는 방법은 서버마다 별도의 가상 지점을 해시 링의 여러 위치에 놓는다.
- 예시로 각 서버에 추가로 가상 지점을 3개 생성한다고 가정했을때 이를 위해 추가 해시함수 h1(), h2(), h3() 3개를 통해 각 서버에 대응하는 추가 지점을 해시 링에 생성해야 한다.
- server0의 경우 해시 함수를 거친 결과는 다음과 같다.
    - h(server0) = s0
    - h1(server0) = s01
    - h2(server0) = s02
    - h3(server0) = s03
- server1은 다음과 같다.
    - h(server1) = s1
    - h1(server1) = s11
    - h2(server1) = s12
    - h3(server1) = s13
- server2와 server3도 같은 방법으로 값을 얻을 수 있다.
- 이 방식의 핵심은 서버의 원래 지점과 가상 지점이 더 고르게 분포해 요청이 서버에 보다 균등하게 할당 된다는 점이다. 한 서버가 다운되더라도 해당 서버에 연결된 지점 4개만 사라지고 다른 서버의 지점은 잘 분산되어 있어 요청이 특정 서버 하나에 집중되지 않고 남아 있는 서버에 거의 균등하게 분배된다. 이는 그림 3-10에서 소개한 일관된 해싱 링의 문제였던 장애 문제를 줄여준다.
- 결론적으로 해시 링으로 데이터 키와 노드 식별자를 콩통 해시 공간에 매핑함으로써 일관된 해싱을  사용하면 데이터를 노드에 일관되게 할당할 수 있고 확장성과 장애 허용성까지 챙길 수 있다. 노드를 추가하거나 삭제할 때 인접한 노드로 일부 데이터만 다시 매핑하면 되기에 데이터 이동량을 제한할 수 있고 시스템이 의도하지 않은 방식으로 동작하는 현상을 막아 준다.
