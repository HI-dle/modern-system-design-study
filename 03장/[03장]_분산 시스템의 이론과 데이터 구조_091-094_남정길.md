# 3.8 하이퍼로그로그 (HyperLogLog)

---

`하이퍼로그로그(HyperLogLog)`는 **집합에서 중복되지 않은 데이터의 개수(Cardinality)** 를 매우 적은 메모리로 추정하기 위한 **확률적 알고리즘**입니다.

이것은 대규모 데이터셋을 다루거나 메모리 효율성이 중요한 경우에 유용하며 집합 크기와 상관없이 고정된 양의 메모리만으로 카디널리티를 근사적으로 계산하여 해시함수와 확률적 카운팅 기법을 활용하여 메모리 절약과 계산 효율성을 높인다.

핵심은 집합의 각 요소를 해싱한후 해시값의 이진표현에서 가장 긴 0의 연속 길이를 찾는것

가장긴 0의 연속길이를 카디널리티의 추정값으로 활용하고 이 추정값들을 평균을 내어 정밀도를 높인다.

예를 들어 웹사이트 방문자 수를 구한다고 가정할때
가장 단순한 방법은 사용자 ID를 저장하고 중복을 제거한 뒤 개수를 세는 방법입니다.

예:

```javascript
const users = new Set();

users.add("john");
users.add("david");
users.add("john");

console.log(users.size); // 2
```

이 방법은 데이터가 많아지면 문제가 발생한다
예를 들어 방문자가 10억 명이라면 사용자 ID를 저장하는 데 필요한 메모리는 방문자 수에 비례해서 증가한다.
만약 사용자 정보를 최소 1바이트로만 표현시 -> 10억명 = 약 1GB 메모리 필요

즉 기존 방식의 문제는 데이터 개수가 증가할수록 메모리 사용량도 선형적으로 증가한다는 것입니다.
이 문제를 해결하기 위해 등장한 것이 바로 **HyperLogLog**!!

HyperLogLog는 데이터를 모두 저장하지 않고도 고유 데이터 개수를 추정할 수 있도록 설계된 알고리즘이다.


데이터를 저장하지 말고 **데이터의 특징만 이용해서 개수를 추정하자**


동전 던지기 예시

HyperLogLog의 핵심 개념을 이해하기 위해 동전 던지기 예시를 보면 동전을 계속 던져서 처음으로 뒷면이 나올 때까지 던진다고 가정

앞면 → 앞면 → 앞면 → 뒷면

이 패턴이 나올 확률은 다음과 같다.

```
1/2 × 1/2 × 1/2 × 1/2 = 1/16
```

즉 이 결과가 나오려면 평균적으로 16번 정도 시도해야 한다고 볼 수 있다

이걸 일반화해서 앞면이 L번 연속으로 나오고 그 다음에 뒷면이 나왔다면:

대략: ```2^(L+1)``` 번 정도 시도했다고 추정이 가능하다

이 의미는 이런 패턴이 나올 정도라면 적어도 16번 정도 시도했을 가능성이 높다는 것
이 개념이 HyperLogLog의 핵심 아이디어!

해시함수로 사용자 이름을 2진수로 변환하고, 이 값이 무작위이고, 또 동일한 사용자는 항상 같은 해시값을 반환한다고 가정

해시함수로 다음과 같은 해시 값을 가진 사용자들이 있을때
```
hash("John_1275") = 111011100110101100101000001100
hash("David.raymond23") = 111011100110101100001000000010
hash("Sarah1978") = 100011100110101100101000000001
hash("John") = 1110111001101011001010000011
```

앞서 동전 던지기예시에서 앞면의 연속(L)대신 해시 값 끝에 있는 0 개수(동전의 뒷면으로 비유)를 사용하여
앞의 사용자 이름 중 끝에 연속된 0의 최대 길이는 2이다. 따라서 대략 2(2+1) = 8명의 방문자가
있다고 추정할 수 있습니다.

| 끝 0 개수 | 확률   |
| ------ | ---- |
| 1      | 1/2  |
| 2      | 1/4  |
| 3      | 1/8  |
| 4      | 1/16 |
| 5      | 1/32 |

즉, 끝에 5개 0이 나오려면 대략 32개 정도 데이터가 있어야 나타날 가능성이 높다.
그래서 0 개수 → 전체 데이터 규모 추정 가능

하지만 문제가 하나 있는데 예를 들어 이런 해시가 나온다면

```
000000000000000000000000000000
```

엄청 큰 값이 되지만 실제 데이터는 적을 수도 있다. 즉, 특정 해시 하나 때문에 결과가 왜곡될 수 있는데
이걸 이상치(outlier) 문제라고 한다.
그래서 정확성을 높이기 위해 다음과 같은 단계를 거친다
1. 해시를 여러 버킷으로 나누기
    ```
    hash → bucket 1
    hash → bucket 2
    hash → bucket 3
    hash → bucket 4
    ```
2. 각 버킷마다 최대 L 저장

    | bucket | max L |
    | ------ | ----- |
    | 1      | 5     |
    | 2      | 3     |
    | 3      | 6     |
    | 4      | 4     |



3. 평균 계산
   - 산술 평균을 쓰지 않고 조화 평균을 사용
   - 산술평균을 사용 했을때
      ```
      (5+3+6+4)/4 = 4.5
      ```
      6 같은 값이 영향 크게 줌 
   -  그래서 사용하는 것이 **조화 평균(harmonic mean)** 이다.
      ```
      N / (1/n1 + 1/n2 + 1/n3 + ...)
      ```
      큰 값 영향을 감소시켜서 이상치 영향을 줄이는 효과가 있다.
   
하이퍼로그로그의 메모리는 얼마나 절약될까?

예:

방문자수 2^(L+1)

```
1,000,000,000억명
```

필요한 최대 L:

```
L = log2(1,000,000,000) = 끝자리가 0인것이 최대 30개
log2(30)= 5비트
```
즉 1 byte면 충분하다. (수백만 배 절약) 이것이 HyperLogLog가 유명한 이유이다.

하이퍼로그로그는 데이터베이스 시스템, 네트워크 트래픽 분석, 웹 분석, 빅 데이터 처리 등에서 중복 없
이 데이터의 실제 개수를 측정하는 데 활용되며 특히 메모리가 제한된 분산 시스템이나 데이터 스트리
밍 환경에서 빠르게 근사치를 뽑아내야 하는 상황에서 힘을 발휘한다.


### 실제 사용 예

Redis:

```
PFADD
PFCOUNT
```

Google BigQuery:

```
APPROX_COUNT_DISTINCT()
```

Apache Spark:

```
approx_count_distinct()
```

대규모 데이터 환경에서는 정확 count보다 HyperLogLog를 더 많이 사용합니다.
정확도보다 속도,메모리,확장성이 더 중요하기 때문

## 3.9 요약

### 1. 분산 시스템 핵심 이론

* **CAP 정리**<br>
  일관성, 가용성, 파티션 허용성 간 상충 관계를 설명하는 이론

* **PACELC 정리**<br>
  네트워크 분할이 없을 때도 일관성과 지연 시간 사이에 선택이 필요하다는 것을 보여주는 정리

* **FLP 불가능성 정리**<br>
  단 하나의 장애가 있는 비동기 시스템에서 합의를 이루는 것에 근본적인 한계가 있음을 의미하는 정리
  (분산 시스템에서 장애 허용성을 보장하는 데 따르는 고유한 어려움)



### 2. 합의 알고리즘

* **Paxos / Raft**<br>
  장애를 견딜 수 있는 분산 시스템에 필수이며 네트워크 분할이나 노드가 정상적으로 동작하지 않는 상황에서도 여러 노드 간 합의를 이끌어 내는 방식

* **비잔티움 장군 문제**<br>
  분산 시스템에서 악의적인 행위자가 일으킬 수 있는 문제를 해결하는 복잡한 메커니즘



### 3. 분산 시스템 자료구조

* **일관적 해싱 (Consistent Hashing)**<br>
  데이터를 여러 노드에 분산하면서도 확장성과 부하 균형을 유지할 수 있는 기법

* **Bloom Filter**<br>
  특정 요소가 집합에 속해 있는지 효율적으로 검사하는 자료구조

* **Count-Min Sketch**<br>
  대규모 분산 시스템에서 발생하는 이벤트 빈도를 근사적으로 계산하는 데 유용한 자료구조

* **HyperLogLog**
  최소한의 메모리로 집합의 고유 개수를 추정할 수 있는 확률적 알고리즘
