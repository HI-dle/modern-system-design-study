# 10.5 요구 사항 검토

음식점 찾기 및 주문하기 서비스의 기능적/비기능적 요구 사항을 제대로 충족하도록 설계했는지 확인해 보겠습니다.

##  기능적 요구 사항

- 사용자는 자신의 위치를 기준으로 주변 음식점을 검색할 수 있어야 합니다.
- 사용자가 음식점을 선택하고 주문할 수 있어야 합니다.

서비스 설계 디자인에 잘 부합.

## 비기능적 요구 사항

- 가용성: 근접 서비스는 항상 높은 가용성을 유지
  - 각 서비스가 여러 인스턴스로 운영되기 때문에 일부 인스턴스가 중단되더라도 시스템 전체는 계속 정상적으로 트래픽을 처리
- 확장성: 사용자 1억 명을 처리할 수 있을 만큼 뛰어난 확장성을 갖추고 있어야 하며, 요청 폭주 상황도 감당 
  - 서비스 내 병목 현상 없이 수평 확장이 가능하도록 설계
- 지연 시간: 읽기 지연 시간은 200밀리초 이내
  - 쿼드 트리를 사용하여 검색 ➡️ 지연 시간 200밀리초 이하 유지
- 일관성: 음식점 데이터가 업데이트될 때 약간의 지연이 발생하지만, 음식점 데이터는 애초에 자주 바뀌지 않으므로 큰 문제없다. 
  - 최종 일관성을 사용하는 것으로도 충분
- 신뢰성: 시스템에 장애가 발생하거나 트래픽이 급증하는 상황에서도 안정적으로 작동
  - 여러 인스턴스를 활용하여 서비스가 중단되지 않도록 설계
  - 데이터의 내구성과 장애 허용성도 갖추고 있어 비기능적 요구 사항을 충분히 충족

# 10.6 요약

- 근접 서비스 이점
  - 위치 데이터를 활용하여 사용자 경험을 높인다. 
  - 서비스 운영을 더 효율적으로 한다. 
  - 여러 산업에서 주목할 만한 성과를 만든다.
- 설계 과정, 기능적 요구 사항과 비기능적 요구 사항을 충족하는 기술적 개념
  - 주변 음식점 검색과 주문하기
  - 가용성, 확장성, 지연 시간, 일관성, 신뢰성 등 비기능적 요구 사항
  - 음식점 검색과 주문을 처리하는 API 설계
  - 트래픽 급증과 사용자 요청을 처리하는 가상의 상황을 예상하며 계산 작업도 수행
- 근접 서비스를 활용하는 다양한 사례
  - 승차 공유, 지역 검색, 음식 배달, 소셜 네트워킹 앱 등.
- '주변 음식점을 찾는 방법'도 해결책과 각 장단점
  - 쿼드 트리
    - 메모리 내에서 빠르게 처리할 수 있고 업데이트가 잦은 경우에도 효율적
  - 지오해시
- 고수준 시스템 아키텍처를 설계
  - 모든 요구 사항을 충족: 가용성, 확장성, 낮은 지연 시간, 최종적 일관성, 신뢰성 등 모든 기능 요구 사항과 비기능 요구 사항을 충족
  - 서비스에 적합한 안정적,효율적인 시스템 완성
